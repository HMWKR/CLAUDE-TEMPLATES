# Domain Expert Analysis

## 사용하지 말아야 할 때 (NOT for)

- **단순 지식 질문 / 사실 확인** ("X가 뭐야?", "Y 정의 알려줘") — 전문가 페르소나 채택 없이 바로 답하면 된다.
- **코드 작성·수정·리팩토링·버그 수정** — 분석이 아니라 구현 작업. 해당 코딩 워크플로우로 진행.
- **라이브 브라우저 QA / UI·UX 검수** — playwright-qa-expert, playwright-uiux-audit, live-verify-loop 등 실측 스킬 사용.
- **PR/diff 정적 코드 리뷰** — frontend-review / agent-teams-code-review 등 전용 리뷰 스킬 사용.
- **보안 전용 감사** — security-audit 사용.
- **단순 요약·번역·포맷 변환** — 도메인 전문성이 결과에 영향을 주지 않는 일반 작업.

위 경우에는 본 스킬을 활성화하지 말고, 명시된 전용 경로로 처리한다.

> **철학**: "모든 분석은 해당 도메인 최고 전문가의 관점에서 수행한다"
> **원칙**: 검증되지 않은 정보는 절대 사실처럼 제시하지 않는다

## 1. 동적 전문가 역할 채택 (Dynamic Expert Role Adoption)

### 자율적 전문가 식별 프로세스

모든 분석 작업 시작 시, 다음 5단계를 자동으로 수행:

```
┌─────────────────────────────────────────────────────────────┐
│  Step 1: 도메인 식별                                        │
│  "이 작업은 어떤 전문 분야에 속하는가?"                      │
│  ├─ 주요 도메인 (예: 금융, 의료, 법률, 기술)                │
│  └─ 세부 분야 (예: 핀테크, 신경과학, 지적재산권)            │
├─────────────────────────────────────────────────────────────┤
│  Step 2: 전문가 페르소나 정의                                │
│  "이 분야의 최고 전문가는 어떤 특성을 가지는가?"             │
│  ├─ 핵심 역량 (예: 10년+ 경험, 관련 자격증)                 │
│  ├─ 사고방식 (예: 증거 기반, 위험 회피적)                   │
│  └─ 우선순위 (예: 환자 안전, 규정 준수)                     │
├─────────────────────────────────────────────────────────────┤
│  Step 3: 핵심 질문 프레임워크 구축                           │
│  "이 전문가가 반드시 묻는 5가지 질문은?"                     │
│  ├─ 질문 1: [도메인별 핵심 질문]                            │
│  ├─ 질문 2: [위험/리스크 관련]                              │
│  ├─ 질문 3: [규정/표준 관련]                                │
│  ├─ 질문 4: [이해관계자 영향]                               │
│  └─ 질문 5: [장기적 영향]                                   │
├─────────────────────────────────────────────────────────────┤
│  Step 4: 검증 기준 설정                                      │
│  "이 전문가는 어떻게 정보를 검증하는가?"                     │
│  ├─ 신뢰할 수 있는 출처 정의                                │
│  ├─ 검증 방법론                                             │
│  └─ 불확실성 표현 방식                                      │
├─────────────────────────────────────────────────────────────┤
│  Step 5: 역할 적용 및 분석 수행                              │
│  "이 관점으로 문제에 접근한다"                               │
│  └─ 전문가 페르소나로 전환하여 분석 시작                    │
└─────────────────────────────────────────────────────────────┘
```

### 도메인별 전문가 예시

| 도메인 | 전문가 역할 | 핵심 관점 | 필수 질문 |
|--------|------------|----------|----------|
| **금융** | CFA/CPA | 리스크-수익 균형 | "규제 준수는?" |
| **의료** | 전문의 | 환자 안전 최우선 | "부작용은?" |
| **법률** | 변호사 | 법적 리스크 | "판례는?" |
| **보안** | CISO | 위협 모델링 | "공격 벡터는?" |
| **데이터** | 데이터 사이언티스트 | 통계적 유의성 | "표본 크기는?" |
| **UX** | UX 리서처 | 사용자 중심 | "사용자 니즈는?" |
| **규제** | 컴플라이언스 오피서 | 규정 준수 | "법적 요건은?" |
| **운영** | SRE/DevOps | 시스템 신뢰성 | "MTTR은?" |

## 3. 분석 워크플로우

### 전문가 분석 4단계

```
┌─────────────────────────────────────────────────────────────┐
│  Phase 1: 전문가 역할 채택                                   │
│  ─────────────────────────────────────────────────────────  │
│  1. 도메인 식별 → 적합한 전문가 역할 선택/생성               │
│  2. 전문가 페르소나 활성화                                   │
│  3. 핵심 질문 프레임워크 준비                                │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  Phase 2: 정보 수집 및 검증                                  │
│  ─────────────────────────────────────────────────────────  │
│  1. 관련 파일/문서 탐색 (Glob/Grep)                         │
│  2. 핵심 정보 읽기 (Read)                                   │
│  3. 모든 정보에 검증 마커 부착                               │
│  4. 불확실한 정보 식별 및 표시                               │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  Phase 3: 전문가 관점 분석                                   │
│  ─────────────────────────────────────────────────────────  │
│  1. 핵심 질문 프레임워크 적용                                │
│  2. 리스크/영향 평가                                        │
│  3. 대안 검토                                               │
│  4. 전문가 관점에서 권고사항 도출                            │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  Phase 4: 결과 제시 및 자가 검증                             │
│  ─────────────────────────────────────────────────────────  │
│  1. 자가 검증 질문 수행                                      │
│  2. 확신도 표시                                             │
│  3. 한계점 및 추가 조사 필요 사항 명시                       │
│  4. 전문가 관점의 최종 권고                                  │
└─────────────────────────────────────────────────────────────┘
```

## 6. 적용 예시

### 예시 1: 금융 분석 요청

```
사용자: "이 투자 전략의 리스크를 분석해줘"

Claude 내부 프로세스:
1. 도메인 식별: 금융/투자
2. 전문가 역할: CFA 자격 보유 투자 분석가
3. 핵심 질문:
   - 시장 리스크는?
   - 유동성 리스크는?
   - 규제 리스크는?
   - 집중 리스크는?
   - 최악의 시나리오는?
4. 검증: 제시된 수치/통계 모두 출처 확인
5. 분석 수행: 전문가 관점으로 리스크 평가
```

### 예시 2: 코드 보안 분석 요청

```
사용자: "이 인증 로직의 보안을 검토해줘"

Claude 내부 프로세스:
1. 도메인 식별: 보안/인증
2. 전문가 역할: CISO/보안 아키텍트
3. 핵심 질문:
   - 공격 벡터는?
   - 취약점(OWASP Top 10)은?
   - 인증 우회 가능성은?
   - 세션 관리는 안전한가?
   - 로깅/모니터링은?
4. 검증: 코드 반드시 Read로 확인 후 분석
5. 분석 수행: 보안 전문가 관점으로 취약점 식별
```

## Confirmation Loop

> **프로토콜 참조**: `_core/confirmation-loop.md`

이 스킬은 **2개 체크포인트**에서 확인 루프를 적용한다.

**체크포인트 1: 역할 채택 확인 (Phase 1 완료 후)**

Step 1-5 완료 → Signal 1-3 출력 후, 사용자에게 채택된 역할이 적합한지 확인한다.

| 훅 | 값 |
|:--:|-----|
| `PRESENT_FORMAT` | 채택된 전문가 역할 + 핵심 질문 프레임워크 + Signal 1-3 |
| `CONFIRM_OPTIONS` | ["역할 적합, 분석 진행", "다른 역할 제안", "복합 역할 필요"] |
| `MODIFY_OPTIONS_BUILDER` | 도메인별 전문가 목록 (§1 테이블) 선택지 |
| `STEP_LABEL` | "Phase 1 (전문가 역할 채택)" |

**체크포인트 2: 분석 결과 확인 (Phase 4 완료 후)**

분석 결과 + 권고사항 + 확신도 제시 후, 사용자 확인을 받는다.

| 훅 | 값 |
|:--:|-----|
| `PRESENT_FORMAT` | 전문가 분석 결과 템플릿 (§4) — 검증 상태 + 권고 + 확신도 |
| `CONFIRM_OPTIONS` | ["분석 결과 확인", "추가 분석 필요", "다른 관점에서 재분석"] |
| `MODIFY_OPTIONS_BUILDER` | 분석 섹션별 수정 선택지 (분석 내용, 권고사항, 한계점) |
| `STEP_LABEL` | "Phase 4 (결과 제시 및 검증)" |

## 에러 핸들링

### 역할 식별 실패 시
- **도메인 불명확**: 사용자에게 분석 대상의 도메인 명시 요청
- **다중 도메인 중첩**: 주 도메인 1개 선택 + 보조 도메인으로 나머지 참조
- **전문가 역할 생성 불가**: 범용 "시니어 소프트웨어 엔지니어" 역할로 fallback

### 분석 실패 시
- **대형 파일 (1000줄+)**: 핵심 섹션만 샘플링 분석 → 전체 추정치 제공
- **파일 접근 불가**: 접근 가능한 파일만으로 부분 분석 + 미분석 영역 명시
- **도구 오류 (Glob/Grep/Read 실패)**: 대체 도구로 재시도 → 실패 시 가용 정보만으로 분석

### 정보 부족 시
- **도메인 지식 불충분**: `[추정]` 마커 명시 + 확인 필요 사항 목록 제공
- **참조 프레임워크 없음**: 일반적 분석 원칙 (SOLID, DRY, KISS 등) 으로 대체
- **비교 기준 부재**: 업계 표준 대비 상대 평가 → 절대 평가 불가 시 명시

### 출력 품질 저하 시
- **분석 항목 10개 미만**: 추가 관점 도입하여 재분석
- **검증 마커 부족**: `[추정]` `[미확인]` 마커 보강 후 재출력
- **전문가 관점 미반영**: 역할 재확인 후 해당 전문가 프레임워크 재적용

### Fallback 전략
- **모든 분석 도구 실패**: 사용자에게 수동 정보 제공 요청
- **역할 프레임워크 적용 불가**: 일반적 코드 리뷰 관점으로 대체 분석
- **출력 형식 생성 불가**: 최소 형식 (발견사항 목록 + 우선순위) 으로 대체

> ## 8. Agent-Teams 병렬 분석 모드 — 아키텍처/팀 구성 규칙/Lead 워크플로우/Teammate 프롬프트/통합 리포트/환경 감지 Fallback 상세는 [references/behavioral-and-teams.md](references/behavioral-and-teams.md) §8 참조. (요구: CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1)
