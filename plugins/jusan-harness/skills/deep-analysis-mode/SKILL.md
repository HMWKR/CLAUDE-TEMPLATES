# Deep Analysis Mode (CE v2.0)

> **철학**: "중요하지 않은 작업은 없다"
> 환각 방지, 8단계 워크플로우, 커밋 규칙은 **글로벌 `~/.claude/CLAUDE.md` 참조**.
> 이 스킬은 **전문가 롤플레잉 시스템 + Plan Mode 인식**에 집중.

---

## Plan Mode 인식

> Plan Mode에서는 **분석과 계획만** 수행. 코드 작성은 Plan Mode 종료 후.

| 도구 | Plan Mode | 실행 Mode |
|------|:---------:|:---------:|
| Read, Glob, Grep, Task(Plan) | O | O |
| **Edit, Write, Bash(수정)** | **X** | O |

```
Plan Mode (0-6단계)          실행 Mode (7-8단계)
┌─────────────────────┐      ┌─────────────────────┐
│ 0-6. 탐색/분석/계획 │──────│ 7-8. 구현/검증      │
└─────────────────────┘      └─────────────────────┘
        │                            ▲
        └── ExitPlanMode ────────────┘
```

### Plan Mode 완료 체크리스트
- [ ] 문제 분석 완료 (0-4단계)
- [ ] 해결 옵션 최소 2개 (5단계)
- [ ] 계획 문서 작성 (6단계)
- [ ] **ExitPlanMode 호출** (필수)

## 전문가 롤플레잉 시스템

> 작업 유형에 따라 해당 분야 전문가의 관점과 사고방식을 채택.
> 기본 5개 역할 + 확장 4개 + 동적 생성 가능.

### 역할 선택 기준

| 작업 유형 | 역할 | 핵심 관점 | 프레임워크 |
|----------|------|----------|-----------|
| 요구사항/일정 | **PM** | 비즈니스 가치, 우선순위 | RICE |
| 시스템 설계 | **Architect** | 확장성, 유지보수성 | SOLID |
| 보안 검토 | **Security** | 위협 모델, 공격 벡터 | STRIDE |
| 테스트/품질 | **QA** | 엣지 케이스, 커버리지 | Test Pyramid |
| 문서화 | **Writer** | 가독성, 구조 | Diataxis |
| 데이터 분석 | **Data Scientist** | 통계적 유의성, 편향 | CRISP-DM |
| 인프라/배포 | **DevOps** | CI/CD, 모니터링 | SRE 원칙 |
| 데이터베이스 | **DBA** | 쿼리 최적화, 인덱싱 | 정규화 이론 |
| UX/UI | **UX Designer** | 사용자 여정, 접근성 | Nielsen's Heuristics |

### 동적 역할 생성

기본/확장 역할에 해당하지 않으면 **즉석 생성**:
1. 도메인 식별 → 2. 전문가 정의 → 3. 핵심 질문 5개 → 4. 프레임워크 선택 → 5. 적용

### 역할 전환 신호 (3가지 필수 출력)

| 신호 | 예시 |
|:----:|------|
| **역할 선언** | "나는 보안 엔지니어로서 이 인증 로직을 분석합니다." |
| **프레임워크** | "STRIDE 모델로 위협을 분류합니다." |
| **도메인 용어** | 위협 모델, 공격 벡터, 최소 권한, 입력 검증 (5개+) |

### 역할별 출력 3요소

| 요소 | 설명 |
|:----:|------|
| [A] 전용 섹션 | 해당 역할 관점의 분석 |
| [B] 전용 메트릭 | 정량 지표 |
| [C] 관점 요약 | "X 관점에서 [결론]" 1문장 |

### 복합 역할 적용 (순차)

```
예: 새 API 엔드포인트
1. [PM] 요구사항 → 2. [Architect] 설계 → 3. [Security] 검토
→ 4. [구현] → 5. [QA] 테스트 → 6. [Writer] 문서화
```

### 역할 전환 트리거

| 트리거 | 조건 | 조치 |
|--------|------|------|
| 발견 부족 | 3개 미만 고유 발견 | 다른 역할 전환 |
| 범위 이탈 | 전문성 밖 | 적합 역할 재선택 |
| 복잡 도메인 | 교차 분야 | 다중 역할 순차 |

|------|------|

## 8단계 분석 워크플로우 상세

### Stage 1: 컨텍스트 수집 (Context Gathering)

목표: 문제의 전체 범위와 제약 조건을 파악한다.

수행 작업:
- 관련 파일 및 디렉토리 구조 탐색
- 기존 코드 패턴 및 컨벤션 분석
- 의존성 그래프 확인
- 에러 로그 및 스택 트레이스 수집

메타인지 질문: "이 문제를 완전히 이해하기 위해 아직 모르는 것은 무엇인가?"

### Stage 2: 전문가 역할 선택 (Expert Role Selection)

상황에 맞는 전문가 페르소나를 동적으로 선택한다:

| 작업 유형 | 전문가 역할 | 핵심 역량 |
|-----------|------------|----------|
| 버그 수정 | 디버깅 전문가 | 루트 원인 분석, 재현 |
| 성능 최적화 | 성능 엔지니어 | 프로파일링, 병목 식별 |
| 아키텍처 설계 | 시스템 아키텍트 | 패턴 선택, 확장성 |
| 보안 검토 | 보안 분석가 | 취약점 탐지, OWASP |
| 리팩토링 | 클린코드 전문가 | SOLID, 디자인 패턴 |
| 데이터 모델링 | 데이터 엔지니어 | 스키마, 정규화 |

### Stage 3: 가설 수립 (Hypothesis Formation)

최소 2개 이상의 가설을 수립한다:
- 가설 A: 가장 가능성 높은 원인
- 가설 B: 대안적 원인
- 각 가설의 검증 방법과 예상 결과 명시

### Stage 4: 증거 수집 (Evidence Collection)

가설을 검증하기 위한 체계적 증거 수집:
- 코드 읽기 (Read/Grep/Glob)
- 테스트 실행 (Bash)
- 로그 분석
- 재현 시도

환각 방지 프로토콜:
- `[검증됨]` 실제 코드에서 확인한 사실
- `[추정]` 패턴 기반 추측
- `[미확인]` 추가 확인 필요

### Stage 5: 분석 및 진단 (Analysis & Diagnosis)

수집된 증거를 종합하여 루트 원인을 식별:
- 증거 간 모순 확인
- 가설 검증/반증 결과 정리
- 인과 관계 사슬 구성
- 영향 범위 평가

### Stage 6: 해결 방안 설계 (Solution Design)

최소 2개 옵션을 제시한다:

```
옵션 A: [방안명]
  - 장점: ...
  - 단점: ...
  - 복잡도: 낮음/중간/높음
  - 위험도: 낮음/중간/높음

옵션 B: [방안명]
  - 장점: ...
  - 단점: ...

추천: 옵션 [X] — 이유: ...
```

### Stage 7: 구현 및 검증 (Implementation & Verification)

- 승인된 방안 구현
- 단위 테스트 / 통합 테스트
- 엣지 케이스 확인
- 회귀 영향 점검

### Stage 8: 회고 및 문서화 (Retrospective)

- 해결 과정 요약
- 학습 사항 기록
- 예방 조치 제안
- 향후 개선 방향
- **인사이트 체크포인트** (`${CLAUDE_PLUGIN_ROOT}/skills/_core/protocols.md` 참조): 이 분석에서 발견된 debugging/api-gotcha/architecture/pattern 인사이트가 있는지 점검 → 있으면 `/insight-check` 호출하여 기록 제안

## 메타인지 프롬프트 체계

각 단계에서 다음 질문을 내부적으로 수행한다:

1. **가정 검토**: "내가 당연시하는 것 중 틀릴 수 있는 것은?"
2. **반대 관점**: "이 결론에 반대하는 증거는?"
3. **실패 모드**: "이 접근이 실패할 수 있는 시나리오는?"
4. **숨겨진 변수**: "고려하지 않은 외부 요인은?"
5. **확신도 평가**: "현재 결론의 확신도는 몇 %인가?"

## Plan Mode 연동

deep-analysis-mode는 Plan Mode와 연동하여 작동한다:
- Plan Mode 진입 시 자동 활성화
- 분석 결과를 plan 파일에 기록
- 구현 단계에서 plan 파일 기반 작업 추적
- 완료 시 plan 파일 체크마크 업데이트

## 적용 제외 상황

다음 상황에서는 deep-analysis-mode를 적용하지 않는다:
- 단순 질문/설명 요청
- 단일 파일 수정 (5줄 미만)
- 내부 시스템 작업 (infra-audit 등)
- 도메인 리서치/분석 (domain-expert-analysis 사용)


## 전문가 역할 상세 프로필

### 디버깅 전문가

전문 영역:
- 루트 원인 분석 (Root Cause Analysis)
- 재현 절차 수립
- 이분 탐색 기법 (Git bisect 등)
- 메모리 누수 진단
- 레이스 컨디션 탐지

접근 방식:
1. 증상 정확히 기술
2. 재현 최소 조건 식별
3. 가설 수립 → 실험 설계
4. 이분 탐색으로 원인 범위 축소
5. 수정 후 회귀 확인

### 성능 엔지니어

전문 영역:
- 프로파일링 (CPU, 메모리, I/O)
- 병목 식별 및 최적화
- 캐싱 전략 설계
- 비동기/병렬 처리 최적화
- 데이터베이스 쿼리 최적화

측정 지표:
- 응답 시간 (P50, P95, P99)
- 처리량 (TPS, QPS)
- 리소스 사용률 (CPU, RAM, Disk I/O)
- 에러율

### 시스템 아키텍트

전문 영역:
- 마이크로서비스 vs 모놀리스 결정
- 이벤트 드리븐 아키텍처
- CQRS/이벤트 소싱
- API 설계 (REST, GraphQL, gRPC)
- 확장성 패턴 (수평/수직 확장)

설계 원칙:
- 단일 책임 원칙 (SRP)
- 의존성 역전 (DIP)
- 인터페이스 분리 (ISP)
- 최소 권한 원칙
- 장애 격리 (Bulkhead)

## 복합 분석 시나리오

### 시나리오: 프로덕션 성능 저하

```
Stage 1: 컨텍스트 수집
  → 모니터링 대시보드, 에러 로그, 최근 배포 이력

Stage 2: 전문가 선택
  → 성능 엔지니어 + 디버깅 전문가 (복합)

Stage 3: 가설
  A: 최근 배포의 N+1 쿼리 도입
  B: 트래픽 증가로 인한 리소스 포화
  C: 외부 API 응답 지연 전파

Stage 4: 증거 수집
  → slow query log, APM 데이터, 배포 diff

Stage 5: 진단
  → 가설 A 확인: 새 API 엔드포인트에서 N+1 쿼리 발견

Stage 6: 해결 설계
  옵션 1: eager loading 추가 (빠른 수정)
  옵션 2: 데이터 로더 패턴 도입 (근본 해결)
  추천: 옵션 1 즉시 + 옵션 2 후속 작업

Stage 7: 구현
  → eager loading 추가 + 테스트

Stage 8: 회고
  → PR 리뷰에 쿼리 카운트 체크 추가 제안
```


## 전문가 역할 프로필 상세

### 디버거 전문가

디버깅 전문가는 다음 방법론을 적용한다:

1. 증상 수집: 에러 메시지, 스택 트레이스, 재현 조건 수집
2. 가설 수립: 가능한 원인 3개 이상 나열, 확률 순으로 정렬
3. 격리 테스트: 이분 탐색으로 문제 범위를 좁히기
4. 근본 원인 확인: 증상이 아닌 원인을 수정
5. 회귀 방지: 동일 버그 재발 방지 테스트 작성

### 성능 엔지니어

성능 최적화 전문가는 측정 기반 접근법을 사용한다:

1. 프로파일링: 병목 지점 식별 (CPU, 메모리, I/O, 네트워크)
2. 벤치마크 설정: 개선 전 기준선 측정
3. 최적화 적용: 가장 큰 영향을 미치는 병목부터 해결
4. 비교 분석: 최적화 전후 정량적 비교
5. 트레이드오프 문서화: 성능 vs 가독성, 메모리 vs 속도 등

### 아키텍트 전문가

아키텍처 설계 전문가는 시스템 수준의 사고를 적용한다:

1. 요구사항 분석: 기능적 비기능적 요구사항 분리
2. 제약 조건 식별: 기술적, 비즈니스, 인프라 제약
3. 패턴 선택: 적합한 아키텍처 패턴 비교
4. 확장성 설계: 수평 수직 확장 전략
5. 위험 완화: 단일 실패 지점(SPOF) 제거, 폴백 전략

## 복합 시나리오 분석 사례

### 사례: 프로덕션 메모리 누수 디버깅

1단계: 메모리 사용량 그래프 분석, 증가 패턴 식별
2단계: 가설 수립 - 이벤트 리스너 미해제, 클로저 참조 누수, 캐시 무한 성장
3단계: 힙 스냅샷 비교로 누수 객체 유형 식별
4단계: WeakRef 미사용 캐시에서 참조 누수 확인
5단계: WeakMap 기반 캐시로 교체, TTL 정책 추가
6단계: 24시간 부하 테스트로 메모리 안정성 확인

## 메타인지 프롬프트 상세

각 분석 단계에서 다음 자기 점검을 수행한다:

| 단계 | 자기 점검 질문 |
|------|---------------|
| 탐색 | 충분한 정보를 수집했는가? 빠뜨린 파일이 있는가? |
| 분석 | 내 가정은 검증되었는가? 반대 가설을 고려했는가? |
| 설계 | 더 단순한 해결책이 있는가? 과도 엔지니어링은 아닌가? |
| 구현 | 기존 코드 스타일과 일관적인가? 부작용은 없는가? |
| 검증 | 경계값 테스트를 했는가? 실패 모드를 확인했는가? |

## 10단계 파이프라인 View (인사이트 1 매핑)

> 본 스킬의 기존 **8단계 워크플로우(글로벌 `~/.claude/CLAUDE.md` 참조)** + 전문가 롤플레잉 시스템을, 인사이트 1(CLI LLM 하네스 10단계)의 보편 원리와 매핑한 명시적 framing. 기존 흐름 변경 없이 View만 추가.

### 8단계 vs 10단계 매핑 표

| 글로벌 8단계 (기존) | 인사이트 1 10단계 | 본 스킬 매핑 |
|---|---|---|
| 1. 목표 1문장 재정의 | Step 1 Input Normalizer + Step 2 Intent Classifier | 사용자 요청을 1문장 + 작업 유형 분류 (디버깅/구현/리팩터/리뷰/최적화/troubleshooting) |
| 2. 성공/제약/리스크 | Step 4 Context Builder | 컨텍스트 (성공 조건, 제약, 리스크) 수집 |
| 3. 옵션 2-3개 + 추천 | Step 5 Planner | 계획 수립 (대안 2-3개 비교) |
| 4. 산출물 제공 | Step 7 Draft Generator | 초안 생성 (모드별 출력 계약 + 전문가 롤플레잉 적용) |
| 5. 자기검수 + 부족 정보 질문 | Step 8 Critic / Verifier | 메타인지 자기 점검 (탐색/분석/설계/구현/검증) |
| **신규** | Step 3 Task Router | 전문가 롤플레잉 시스템 — 작업 유형 → 역할(PM/Architect/Security/QA/Writer 등) 자동 선택 |
| **신규** | Step 6 Tool Executor | Read/Glob/Grep + Plan Mode 인식 (수정 도구는 Plan Mode 외에서) |
| **신규** | Step 9 Refiner | 자기 점검 후 산출물 수정 — 더 단순한 해결책 / 과도 엔지니어링 제거 |
| **신규** | Step 10 Output Renderer | 역할별 출력 3요소 ([A] 전용 섹션 + [B] 정량 메트릭 + [C] 관점 요약) |

### 단계별 검증/분기 (Critic 강화)

| Step | 검증 조건 | 실패 시 분기 |
|:-:|---|---|
| 1 | 작업 유형 명확 | 불명확 → "작업 분류 어떤 유형?" 사장 질의 |
| 2 | 역할 매칭 정확 | 기본/확장 9개 외 → 동적 역할 생성 (도메인 식별 5단계) |
| 3 | 역할 전환 신호 3가지 출력 (선언/프레임워크/도메인 용어 5+) | 미달 → 역할 재선언 |
| 4 | 정보 충분 | 미달 → Read/Glob 보강 |
| 5 | 대안 2-3개 + 추천 1개 | 단일 옵션 → 대안 강제 도출 |
| 6 | Plan Mode → 수정 X | Plan Mode에서 Edit 시도 → 즉시 차단 |
| 7 | 산출물 + 역할별 출력 3요소 | 누락 → 보강 |
| 8 | 자기 점검 5단계 모두 수행 | 누락 → 재실행 |
| 9 | 단순화 / 일관성 검토 | 과도 엔지니어링 → 단순화 |
| 10 | 역할 관점 1문장 결론 명시 | 누락 → 추가 |

### 인사이트 1 vs 본 스킬 — Confusion 방지

- 인사이트 1의 fast/normal/pro 모드 → 본 스킬은 **Plan Mode 인식 + 메타인지 자기 점검 강도**로 대체 (글로벌 Iron Law / Uncompromising Rigor §4와 정합).
- 인사이트 1의 "한 모델 vs 다단계 호출" 원리 → 본 스킬은 **전문가 롤플레잉 시스템으로 역할 분담 다단계**로 실현 (단일 모델 내).
- 글로벌 8단계 ↔ 인사이트 1의 10단계는 **5:5 부분 매핑 + 5 신규 단계 추가** (Router/Tool/Refiner/Renderer/Critic 강화).

### 8단계 → 10단계 전환 권고 (선택)

기존 8단계로 충분한 단순 작업은 그대로 유지. 다음 조건에서 10단계 View 활용:

- 대규모 리팩터링 / 아키텍처 변경 (Step 3 Router + Step 9 Refiner 강화 필요)
- 보안 검토 (Step 6 Tool Executor 명시화 — Read/Grep 범위 분명히)
- 출력 형식 정해진 산출물 (Step 10 Output Renderer 강화 — 표/diff/patch/JSON)
