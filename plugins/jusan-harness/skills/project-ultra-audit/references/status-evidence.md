## §7 테스트 상태 표기 규칙 (5개만 사용)

| 상태 | 의미 | 사용 조건 |
|---|---|---|
| PASS | 실제 실행했고 기대=실제 일치 | 증거 필수 |
| FAIL | 실제 실행했지만 기대≠실제 | 재현 절차 필수 |
| BLOCKED | 계정·권한·환경·서버·데이터 문제로 막힘 | 필요 조건 명시 |
| NOT TESTED | 아직 테스트 안 함 | PASS처럼 표현 금지 |
| NOT APPLICABLE | 해당 프로젝트에 미적용 | 이유 명시 |

**최종 판정으로 금지**: OK / 완료 / 확인됨 / 문제없음 / 대체로 가능 / 정상으로 보임.
→ rigor §2 자기정당화 훅이 이 표현들을 차단한다.

## §10 실제 작동 확인 계약
기능이 "작동한다"고 말하려면 §3.1 증거 중 하나 이상 필요. 확인 결과는 다음 형식:
- 확인한 기능 / 확인 방법 / 사용한 계정·권한 / 입력값 / 실행 결과 / 기대 결과와 일치 여부 / 증거 / 남은 문제.
증거 없으면 "작동 확인 완료"라고 말하지 않는다.

## §11 실행 환경 없을 때 응답 규칙
실행 환경·코드·계정·서버·테스트 도구·DB 접근이 없어 실제 검증 불가 시 반드시 다음 문장 포함:

```text
현재는 실제 실행 환경이 제공되지 않았기 때문에 실제 작동 여부를 확인할 수 없습니다. 따라서 아래 항목은 설계상 테스트 계획이며, 작동 확인 완료로 표시하지 않습니다.
```

그 다음 작성: 실제 확인에 필요한 자료 / 필요한 계정 / 필요한 실행 환경 / 필요한 명령어 / 필요한 접근 권한 / 확인해야 할 URL / 확인해야 할 데이터 / 실제 테스트 순서 / 현재 BLOCKED 항목 / 현재 NOT TESTED 항목.
