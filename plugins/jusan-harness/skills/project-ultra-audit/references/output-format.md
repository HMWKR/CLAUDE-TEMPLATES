## §8 테스트 케이스 출력 계약 (13컬럼 표)

| ID | 구분 | 페이지 | 기능 | 사용자 역할 | 사전 조건 | 실행 절차 | 입력값 | 기대 결과 | 실제 결과 | 상태 | 증거 | 비고 |
|---|---|---|---|---|---|---|---|---|---|---|---|---|

상태는 PASS/FAIL/BLOCKED/NOT TESTED/NOT APPLICABLE 중 하나. 실제로 실행 안 했으면 PASS가 아니라 NOT TESTED.

### test_type (참고)
normal / exception / failure / edge_case / permission / state_transition / input_validation / notification / admin_reflection / log_audit / recovery / concurrency / duplicate_submission / accessibility / responsive / performance / integration.

## §9 권한 테스트 계약 (필수 포함)
검토 조합: 비회원 / 로그인 사용자 / 권한 없는 사용자 / 정지된 사용자 / 탈퇴한 사용자 / 일반 관리자 / 슈퍼 관리자 접근 시 × URL 직접 입력 / API 직접 호출 / 타인 데이터 요청 / 삭제된 데이터 접근.

## §13 완료 기준 (모두 만족해야 완료 — 증거 없으면 완료 선언 금지)
모든 공개·사용자·관리자 페이지 정상 표시 / 모든 필수 기능 실제 작동 / 모든 권한 제한 정상 작동 / 모든 입력값 검증 정상 / 모든 데이터 저장·수정·삭제 정상 / 새로고침 후 데이터 유지 / 관리자에서 데이터 정상 관리 / 사용자 페이지 데이터 정상 반영 / 오류 상황 적절한 메시지 / 모바일 주요 기능 사용 가능 / 테스트 결과 문서화 / 실패 테스트 안 숨김 / 미확인 항목 명확 표시.

## §15 최종 자체 점검 (답변 전 — 12항목)
관리자 페이지 빠짐없이 검토? / 모든 사용자 역할 검토? / 모든 페이지 나열? / 모든 버튼·입력창 검토? / 권한 없는 접근 테스트 항목 포함? / 데이터 저장·수정·삭제 확인 방법 기재? / 실제 미확인을 PASS로 안 썼나? / 추정·확정 구분? / 실패 가능성 안 숨김? / 완료 기준 명확? / 미확인 항목 분리? / BLOCKED 조건 명시? — 하나라도 부족하면 수정 후 제출.
