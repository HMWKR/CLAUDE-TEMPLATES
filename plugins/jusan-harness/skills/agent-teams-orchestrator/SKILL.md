## 1. 작업 분석 & 오케스트레이션 패턴 자동 선택

### 5가지 오케스트레이션 패턴

| 패턴 | 적합한 상황 | 팀원 수 | 통신 방식 |
|------|-----------|---------|----------|
| **Parallel Specialists** | 독립적 작업 3개 이상 | 작업당 1명 | 최소 (결과만 보고) |
| **Pipeline** | 순차 의존성 있는 작업 | 단계당 1명 | 순차 전달 |
| **Research + Implementation** | 조사 후 구현 | 2-3명 | 연구→구현 핸드오프 |
| **Plan Approval** | 설계 검토 필요 | 2-4명 | 계획→승인→구현 |
| **Swarm** | 대규모 복잡한 작업 | 4-6명 | 자유 통신 |

### 100점 자동 스코어링 (패턴 선택 의사결정)

작업 요청을 받으면 다음 5개 차원으로 자동 스코어링하여 최적 패턴을 선택한다:

```
┌─────────────────────────────────────────────────────────────┐
│  차원 1: 작업 독립성 (30점)                                  │
│  ├─ 30점: 완전 독립 (파일/모듈 겹침 없음)                   │
│  ├─ 20점: 대부분 독립 (일부 공유 인터페이스)                 │
│  ├─ 10점: 부분 의존 (공유 상태/파일 있음)                    │
│  └─  0점: 밀결합 (동일 파일/함수 수정 필요)                  │
├─────────────────────────────────────────────────────────────┤
│  차원 2: 순차 의존성 (20점)                                  │
│  ├─ 20점: 완전 순차 (A→B→C 필수)                            │
│  ├─ 10점: 부분 순차 (일부만 순서 필요)                       │
│  └─  0점: 순서 무관 (모두 병렬 가능)                         │
├─────────────────────────────────────────────────────────────┤
│  차원 3: 리스크/복잡도 (20점)                                │
│  ├─ 20점: 매우 높음 (아키텍처 변경, 보안 영향)               │
│  ├─ 15점: 높음 (다수 모듈 영향)                              │
│  ├─ 10점: 중간 (단일 기능 구현)                              │
│  └─  5점: 낮음 (단순 수정)                                   │
├─────────────────────────────────────────────────────────────┤
│  차원 4: 예상 작업 규모 (15점)                               │
│  ├─ 15점: XL (파일 10+, 아키텍처 변경)                       │
│  ├─ 10점: L (파일 5-10개, 기능 다수)                         │
│  ├─  5점: M (파일 3-5개, 기능 1개)                           │
│  └─  2점: S (파일 1-2개, 단순 수정)                          │
├─────────────────────────────────────────────────────────────┤
│  차원 5: 파일 충돌 가능성 (15점)                             │
│  ├─ 15점: 없음 (완전 분리된 파일)                            │
│  ├─ 10점: 낮음 (공유 파일 1-2개)                             │
│  ├─  5점: 중간 (공유 파일 3개 이상)                          │
│  └─  0점: 높음 (동일 파일 동시 수정)                         │
└─────────────────────────────────────────────────────────────┘
```

### 스코어 → 패턴 매핑

```
총점 계산 후 패턴 자동 선택:

독립성 높음(25+) + 순차성 낮음(0-5) + 충돌 낮음(10+)
  → Parallel Specialists

순차성 높음(15+) + 독립성 낮음(0-10)
  → Pipeline

리스크 높음(15+) + 규모 중간(5-10)
  → Plan Approval

독립성 중간(10-20) + 리스크 중간(10-15)
  → Research + Implementation

규모 XL(15) + 리스크 높음(15+) + 팀원 4명 이상 필요
  → Swarm
```

### 패턴 선택 예시

```
예시 1: "독립적인 버그 3건 동시 수정"
  독립성: 30  순차성: 0  리스크: 5  규모: 5  충돌: 15
  → Parallel Specialists (팀원 3명, 각자 버그 1건)

예시 2: "API 설계 → 구현 → 테스트"
  독립성: 0  순차성: 20  리스크: 15  규모: 10  충돌: 5
  → Pipeline (Architect → Developer → QA)

예시 3: "인증 시스템 전면 교체"
  독립성: 10  순차성: 10  리스크: 20  규모: 15  충돌: 5
  → Swarm (Architect + Security + Backend + Frontend + QA)
```

## 2. 동적 팀 구성 (domain-expert-analysis 확장)

### 단일 에이전트 → 멀티 에이전트 팀 확장

```
domain-expert-analysis (기존)          agent-teams-orchestrator (신규)
─────────────────────────              ─────────────────────────────
1명의 에이전트가                        Lead가 도메인 분석 후
도메인별 전문가 역할 채택    →          각 전문가를 독립 팀원으로 생성

예: "보안 검토"                         예: "보안 검토"
  → 1명이 CISO 역할                     → 팀원A: CISO (위협 모델링)
                                        → 팀원B: AppSec (코드 취약점)
                                        → 팀원C: QA (보안 테스트)
```

### 기본 역할 카탈로그 (8개)

| # | 역할 | 전문가 페르소나 | 핵심 관점 | 필수 질문 |
|:-:|------|---------------|----------|----------|
| 1 | **PM** | 프로덕트 매니저 | 요구사항 충족, 사용자 가치 | "사용자에게 어떤 가치?" |
| 2 | **Architect** | 소프트웨어 아키텍트 | 구조적 일관성, 확장성 | "기존 아키텍처와 정합?" |
| 3 | **Security** | CISO / AppSec | 위협 모델링, OWASP Top 10 | "공격 벡터는?" |
| 4 | **QA** | QA 리드 / SDET | 엣지 케이스, 회귀 방지 | "실패 시나리오는?" |
| 5 | **Frontend** | React 시니어 개발자 | 컴포넌트 설계, A11y, 성능 | "접근성은 충족?" |
| 6 | **Backend** | 백엔드 아키텍트 | API 설계, 데이터 흐름 | "확장성은?" |
| 7 | **DevOps** | SRE / DevOps | 배포, 모니터링, 안정성 | "MTTR은?" |
| 8 | **Writer** | 테크니컬 라이터 | 문서화, 명확성 | "다른 개발자가 이해 가능?" |

### 동적 역할 생성

기본 8개 역할으로 충분하지 않은 경우, 작업 도메인에 따라 자동 생성한다:

```
┌─────────────────────────────────────────────────────────────┐
│  동적 역할 생성 5단계 (domain-expert-analysis 패턴 차용)      │
├─────────────────────────────────────────────────────────────┤
│  Step 1: 도메인 식별                                        │
│  "이 작업은 기본 8역할로 충분한가?"                          │
│  └─ 부족 시 → 세부 도메인 식별                              │
├─────────────────────────────────────────────────────────────┤
│  Step 2: 전문가 페르소나 정의                                │
│  "이 도메인의 최고 전문가 특성은?"                           │
│  ├─ 핵심 역량                                               │
│  ├─ 사고방식                                                │
│  └─ 우선순위                                                │
├─────────────────────────────────────────────────────────────┤
│  Step 3: 핵심 질문 프레임워크                                │
│  "이 전문가가 반드시 묻는 5가지 질문은?"                     │
├─────────────────────────────────────────────────────────────┤
│  Step 4: 팀원 프롬프트 생성                                  │
│  → 섹션 4의 프롬프트 템플릿으로 자동 구성                    │
├─────────────────────────────────────────────────────────────┤
│  Step 5: spawnTeam으로 팀원 생성                             │
│  → Lead가 TeammateTool로 팀원 배포                           │
└─────────────────────────────────────────────────────────────┘
```

### 동적 역할 예시

| 작업 도메인 | 동적 역할 | 전문가 페르소나 | 핵심 질문 |
|------------|----------|---------------|----------|
| 금융 | CFA 분석가 | 리스크-수익 균형 | "규제 준수는?" |
| 의료 | 의료정보학 전문의 | 환자 안전 최우선 | "부작용은?" |
| 데이터 | 데이터 사이언티스트 | 통계적 유의성 | "표본 크기는?" |
| 성능 | 성능 엔지니어 | 측정 기반 최적화 | "Profiling 결과는?" |
| 접근성 | A11y 전문가 | WCAG 2.2 준수 | "스크린 리더 호환?" |
| 국제화 | i18n 엔지니어 | 다국어 지원 | "RTL 레이아웃 필요?" |

## 3. Lead 역할 가이드

### Lead 세션 6단계 워크플로우

```
┌─────────────────────────────────────────────────────────────┐
│  Lead 세션 워크플로우                                        │
│                                                             │
│  [1] 작업 분석                                              │
│      ├─ 사용자 요청 파악                                    │
│      ├─ 100점 스코어링으로 패턴 선택                        │
│      └─ 산출물: Goal Statement + 패턴 + 스코어              │
│                         ↓                                   │
│  [2] 팀 구성                                                │
│      ├─ 역할 카탈로그에서 필요 역할 선택                    │
│      ├─ 부족 시 동적 역할 생성                              │
│      └─ 산출물: 팀원 목록 + 역할 배정                       │
│                         ↓                                   │
│  [3] 팀원 생성 (spawnTeam)                                  │
│      ├─ 섹션 4의 프롬프트 템플릿으로 각 팀원 생성           │
│      ├─ TaskCreate로 전체 태스크 목록 생성                  │
│      └─ 산출물: 활성 팀원 + 태스크 리스트                   │
│                         ↓                                   │
│  [4] 작업 분배 & 조율                                       │
│      ├─ message: 개별 팀원에게 작업 지시                    │
│      ├─ broadcast: 전체 상태 공유                           │
│      ├─ approvePlan: 팀원 계획 검토/승인                    │
│      └─ 산출물: 진행 상태 추적                              │
│                         ↓                                   │
│  [5] 품질 게이트 (섹션 6)                                   │
│      ├─ 팀원 결과 검증                                      │
│      ├─ 문제 발견 시 message로 수정 지시                    │
│      └─ 산출물: 검증 완료 결과                              │
│                         ↓                                   │
│  [6] 통합 & 종료                                            │
│      ├─ 최종 결과 병합                                      │
│      ├─ requestShutdown으로 팀원 정리                       │
│      └─ 산출물: 최종 산출물 + 검증 보고서                   │
└─────────────────────────────────────────────────────────────┘
```

### Delegate Mode 활용

Lead 세션은 **Shift+Tab**으로 위임 모드에 진입하여 조율에 집중할 수 있다:

```
┌─ Delegate Mode (Lead) ──────────────────────────────────────┐
│                                                             │
│  ✅ 허용:                                                   │
│  ├─ message/broadcast로 팀원 조율                           │
│  ├─ TaskCreate/TaskUpdate로 작업 관리                       │
│  ├─ approvePlan으로 팀원 계획 승인                          │
│  ├─ Read/Glob/Grep으로 코드 탐색 (읽기 전용)               │
│  └─ requestShutdown으로 팀원 정리                           │
│                                                             │
│  ❌ 금지 (팀원에게 위임):                                   │
│  ├─ Edit/Write로 코드 수정                                  │
│  ├─ Bash로 명령 실행                                        │
│  └─ 직접 구현 작업                                          │
│                                                             │
│  원칙: "Lead는 지휘자, 팀원은 연주자"                       │
└─────────────────────────────────────────────────────────────┘
```

## 4. 팀원 Spawn 프롬프트 템플릿

### 기본 템플릿 구조 (4-Block)

팀원 생성 시 반드시 다음 **4개 블록**으로 프롬프트를 구성한다:

```markdown
=== Block 1: Context Priming ===

당신은 {팀명}의 Teammate {N}입니다.
Lead가 수집/준비한 다음 데이터를 기반으로 작업합니다:

**프로젝트 개요**:
- 프로젝트: {프로젝트명}
- 기술 스택: {주요 기술}
- 주요 규칙: {CLAUDE.md 핵심 요약}

**현재 작업 목표**:
- 전체 목표: {사용자 요청 요약}
- 오케스트레이션 패턴: {선택된 패턴}
- 팀 구성: {전체 팀원 목록과 역할}

**참조 데이터 파일**:
- {data-file-1.md}: {설명}
- {data-file-2.md}: {설명}

**이 팀원의 담당 범위**:
- 담당 영역: {구체적 파일/모듈/기능}
- 다른 팀원과의 경계: {겹치지 않는 범위}

=== Block 2: Role Definition ===

나는 {N}년 경력의 {역할명}으로서 {핵심 미션 1문장}.

**Signal 1**: "나는 {직함}로서 {핵심 행동 선언}."
**Signal 2**: "적용 프레임워크: {프레임워크1}, {프레임워크2}, {프레임워크3}"
**Signal 3**: 필수 전문 용어 — {용어1}, {용어2}, {용어3}, {용어4}, {용어5}, {용어6}, {용어7}

> Signal 1: 역할 선언문 — 전문가 페르소나 활성화
> Signal 2: 프레임워크 참조 — 도메인 지식 활성화
> Signal 3: 전문 용어 7개+ — 분석 깊이 보장

**전문성**:
1. {핵심 역량 1}
2. {핵심 역량 2}
3. {핵심 역량 3}

**제약**:
1. {역할 범위 제한 1}
2. {역할 범위 제한 2}
3. {역할 범위 제한 3}

**핵심 질문 프레임워크**:
이 역할에서 반드시 점검해야 할 5가지 질문:
1. {도메인별 핵심 질문}
2. {위험/리스크 관련}
3. {규정/표준 관련}
4. {이해관계자 영향}
5. {장기적 영향}

=== Block 3: Task Instructions ===

### 규모별 팀 구성 예시

**S (소규모)**: 독립 버그 2건 동시 수정
```
Lead → spawnTeam([
  { role: "Frontend", task: "버그 #1 - 버튼 클릭 이벤트" },
  { role: "Backend",  task: "버그 #2 - API 응답 파싱" }
])
패턴: Parallel Specialists
```

**M (중규모)**: 새 컴포넌트 추가
```
Lead → spawnTeam([
  { role: "Architect",  task: "컴포넌트 설계 및 인터페이스 정의" },
  { role: "Frontend",   task: "컴포넌트 구현" },
  { role: "QA",         task: "테스트 작성" }
])
패턴: Research + Implementation
흐름: Architect(설계) → Frontend(구현) + QA(테스트 병렬)
```

**L (대규모)**: 상태 관리 리팩토링
```
Lead → spawnTeam([
  { role: "Architect",  task: "새 상태 구조 설계" },
  { role: "Frontend-A", task: "컴포넌트 A-F 마이그레이션" },
  { role: "Frontend-B", task: "컴포넌트 G-L 마이그레이션" },
  { role: "QA",         task: "회귀 테스트" }
])
패턴: Pipeline → Parallel
흐름: Architect(설계 승인) → Frontend-A + Frontend-B(병렬) → QA(검증)
```

**XL (초대규모)**: 인증 시스템 전면 교체
```
Lead → spawnTeam([
  { role: "Architect", task: "아키텍처 설계" },
  { role: "Security",  task: "위협 모델링 및 보안 검토" },
  { role: "Backend",   task: "인증 API 구현" },
  { role: "Frontend",  task: "로그인/회원가입 UI" },
  { role: "QA",        task: "보안 테스트 + E2E" },
  { role: "Writer",    task: "API 문서 + 마이그레이션 가이드" }
])
패턴: Swarm
흐름: 자유 통신, Lead가 교통 정리

## 6. 행동 채택 표준 (Behavioral Adoption Standards)

> **전문가 역할 정의**: `~/.claude/skills/_core/roles.md` 참조
> **Agent-Teams 패턴**: `~/.claude/skills/_core/team-patterns.md` 참조

### 이 스킬의 오케스트레이터 역할

오케스트레이터는 **역할 채택 표준의 조율자**로서, 각 팀원 Spawn 시 `_core/roles.md`에 정의된 Signal 1-3, 출력 형식 [A][B][C], 역할 전환 프로토콜을 Spawn 템플릿에 자동 통합한다.

**오케스트레이터의 책임**:
1. **역할 선택**: 작업 분석 후 적절한 전문가 역할 자동 선택 (기존 스킬 역할 재사용 우선)
2. **Spawn 템플릿 생성**: 선택된 역할에 `_core/roles.md`의 행동 채택 표준 통합
3. **검증 자동화**: 게이트 1-3에서 행동 채택 표준 준수 여부 자동 검증

### 역할 재사용 맵

## 8. 멀티 에이전트 환각 방지 프로토콜

> **공통 프로토콜**: `~/.claude/skills/_core/protocols.md` 참조

**이 스킬 고유의 멀티 에이전트 확장 규칙**:

### 8.1 교차 팀원 검증 (Cross-Teammate Verification)

팀원 A의 산출물을 팀원 B가 사용할 때:
1. 팀원 A가 Lead에게 결과 보고 (마커 포함)
2. Lead가 결과물의 `[검증됨]` 마커 확인
3. `[추정]`/`[미확인]` 항목 → Lead가 직접 검증 or 팀원 A에게 재검증
4. 검증 완료 후에만 팀원 B에게 전달
5. 팀원 B도 핵심 사항은 독립적으로 재확인

**금지**: 팀원 A의 `[추정]` 정보를 팀원 B에게 `[검증됨]`으로 전달 → **L1 치명 환각**

**Pipeline 패턴 주의**: 순차 의존성에서 미검증 정보 전파 시 전체 파이프라인 오염. 각 단계 전환 시 Lead가 반드시 마커 상태 확인.

### 8.2 Lead 검증 책임

- L1/L2 발견 시 → 해당 팀원에게 message로 즉시 재작업 지시
- L3/L4 → 최종 통합 시 Lead가 보완 처리
- 팀원 보고에 마커 누락 시 → message로 마커 추가 요청

## 9. 통신 프로토콜

### 4가지 통신 유형

| 유형 | TeammateTool 연산 | 사용 시점 | 예시 |
|------|-------------------|----------|------|
| **message** (1:1) | `write` | 특정 팀원에게 지시/피드백 | "Frontend: 인증 컴포넌트 검토 결과 보내줘" |
| **broadcast** (전체) | `write` (broadcast) | 전체 상태 공유/방향 전환 | "전원: API 스펙 변경됨, 새 스펙 확인" |
| **approvePlan** | `approvePlan` | 팀원 계획 검토 후 승인 | 팀원의 구현 계획 검토 → 승인/반려 |
| **requestShutdown** | `requestShutdown` | 작업 완료/취소 | 팀원 정리 종료 |

### 통신 시나리오별 가이드

**시나리오 1: 작업 분배**
```
Lead → broadcast: "작업 시작합니다. 각자 담당 확인 후 계획 수립해주세요."
팀원A → Lead: "인증 모듈 구현 계획을 수립했습니다. 승인 요청드립니다."
Lead → approvePlan(팀원A): 계획 승인
팀원B → Lead: "테스트 계획을 수립했습니다."
Lead → approvePlan(팀원B): 계획 승인
```

**시나리오 2: 방향 전환**
```
Lead → broadcast: "요구사항이 변경되었습니다. 모든 작업 일시 중지."
Lead → message(팀원A): "인증 방식을 JWT에서 세션으로 변경해주세요."
Lead → message(팀원B): "테스트 케이스를 세션 기반으로 수정해주세요."
```

**시나리오 3: 문제 해결**
```
팀원A → Lead: "파일 충돌 발생. types/models.ts를 팀원B도 수정 중입니다."
Lead → message(팀원B): "types/models.ts 수정을 중지하고, 팀원A 완료 후 진행."
Lead → message(팀원A): "models.ts 수정을 먼저 완료해주세요. 팀원B가 대기 중."
```

**시나리오 4: 작업 완료**
```
Lead: 모든 게이트 통과 확인
Lead → broadcast: "모든 작업이 완료되었습니다. 종료합니다."
Lead → requestShutdown(팀원A)
Lead → requestShutdown(팀원B)
Lead → requestShutdown(팀원C)
Lead: 최종 결과 사용자에게 보고
```

### 통신 규칙

```
┌─────────────────────────────────────────────────────────────┐
│  필수 규칙                                                   │
│                                                             │
│  1. 팀원 간 직접 통신 금지                                   │
│     → 모든 조율은 Lead를 경유                                │
│                                                             │
│  2. 상태 변경 시 broadcast 필수                              │
│     → 요구사항 변경, 스펙 변경, 일정 변경                    │
│                                                             │
│  3. 파일 충돌 감지 시 즉시 보고                              │
│     → Lead가 순서 조정                                       │
│                                                             │
│  4. 작업 완료 시 Lead에게 결과 보고 필수                     │
│     → 완료 항목, 발견 사항, 잔여 리스크 포함                 │
│                                                             │
│  5. 계획 변경 시 approvePlan 재요청 필수                     │
│     → 승인 없이 계획 변경 금지                               │
│                                                             │
│  6. 모든 message/broadcast에 검증 마커 필수                  │
│     → 코드/파일 관련 정보: [검증됨] 또는 [추정] 필수         │
│     → API/라이브러리 정보: [문서확인] 필수                    │
│     → 마커 없는 기술 정보 전달 금지                           │
│                                                             │
│  7. 다른 팀원의 결과물 인용 시 출처 팀원 명시                │
│     → 예: "[검증됨, Architect 확인] utils/unitEngine.ts:245" │
│     → [추정] 정보를 [검증됨]으로 전달 시 L1 치명 환각        │
└─────────────────────────────────────────────────────────────┘
```

## 11-1. 의미적 루프 방지 프로토콜 (Semantic Loop Prevention)

> **핵심 원칙**: 세션 간 반복(컨텍스트 소진 → 재시작 → 동일 작업 반복)을 방지한다.

### 반복 감지 프로토콜

Lead는 팀 작업 시작 전 반드시 다음을 확인:

```
┌─────────────────────────────────────────────────────────────┐
│  작업 시작 전 반복 감지 체크리스트                            │
│                                                             │
│  1. 기존 결과물 확인                                         │
│     ├─ Glob으로 qa-data/, output/, results/ 등 확인          │
│     ├─ checkpoint.json 존재 여부 확인                        │
│     └─ summary.md 등 완료 표시 파일 확인                     │
│                                                             │
│  2. 감지 결과별 행동                                         │
│     ├─ 결과물 존재 + 완료 → 재수집 안함, 기존 데이터 활용   │
│     ├─ 결과물 존재 + 미완료 → checkpoint에서 이어서 진행    │
│     └─ 결과물 없음 → 정상적으로 새 작업 시작                │
│                                                             │
│  3. 팀원에게 전달                                            │
│     └─ "기존 데이터가 있으면 재수집하지 말 것" 지침 필수     │
└─────────────────────────────────────────────────────────────┘
```

### 컨텍스트 예산 관리

| 단계 | 예산 | 초과 시 행동 |
|:----:|:----:|-------------|
| Stage 1 (수집) | 30% 이내 | checkpoint 저장 후 즉시 Stage 2로 전환 |
| Stage 2 (분석) | 50% 이내 | 분석 우선순위 높은 항목만 처리 |
| Stage 3 (보고) | 20% 이내 | 핵심 발견만 요약 보고 |

### 체크포인트 프로토콜

Lead는 각 Stage 완료 시 체크포인트를 디스크에 저장:

```json
{
  "task": "작업명",
  "stage": 1,
  "completed_steps": [1, 2, 3],
  "last_step": 3,
  "timestamp": "ISO-8601",
  "pages_visited": [],
  "status": "in_progress"
}
```

## 12. 주의사항 (Red Flags)

### 절대 금지

- ❌ 게이트 검증 없이 완료 선언
- ❌ 팀원 간 직접 통신 (Lead 우회)
- ❌ 동일 파일 동시 수정 (파일 충돌)
- ❌ 승인 없는 계획 변경
- ❌ 환각 방지 프로토콜 무시
- ❌ 검증 마커 없이 정보 제시
- ❌ 팀원에게 CLAUDE.md 규칙 미전달

### 패턴 선택 오류 방지

```
단일 작업 → Agent Teams 불필요 (일반 모드 사용)
팀원 1명  → Task 도구 서브에이전트로 충분
팀원 7명+ → 관리 오버헤드 과다 (최대 6명 권장)
```

### 팀원 문제 대응

```
팀원이 응답하지 않음 → Escape로 인터럽트 후 재지시
팀원이 영역 침범    → message로 즉시 중단 지시
팀원 간 충돌 발생   → Lead가 우선순위 결정 후 조정
```

---

> **최종 원칙**: "Lead는 조율자, 팀원은 전문가, 품질은 전원의 책임"

## 10단계 파이프라인 View (인사이트 1 매핑, 2026-05-25 P1-4 추가)

> 본 스킬은 인사이트 2의 자재(Subagents / 분담)를 가장 강하게 활용하는 스킬이며, 인사이트 1(CLI LLM 하네스 10단계)의 보편 원리를 **다중 에이전트 조율** 형태로 실현한다. 본 섹션은 명시적 framing — 기존 본문 변경 없이 View만 추가.

### 다중 에이전트 조율 = 인사이트 1의 다단계 호출

| Step | 인사이트 1 단계 | 본 스킬 매핑 |
|:-:|---|---|
| **1** | Input Normalizer | 사용자 요청 정규화 — 팀 구성 필요 여부 / 도메인 식별 |
| **2** | Intent Classifier | 작업 유형 분류 — Agent Teams 필요 (단일 작업 X, 1명 팀원 X, 7명+ X) |
| **3** | Task Router | **Lead가 팀원 분배** — 영역별 전문가 (FE/BE/AI/QA/Security/Design 등) |
| **4** | Context Builder | 공유 컨텍스트 작성 — CLAUDE.md / 도메인 자료 / 제약 / 출력 형식 |
| **5** | Planner | Lead가 계획 수립 + 각 팀원 작업 영역 + 의존성 표 |
| **6** | Tool Executor | 각 팀원이 자기 영역 도구 실행 (Read/Edit/Write/MCP/Bash) — 병렬 |
| **7** | Draft Generator | 각 팀원이 자기 산출물 작성 (별도 컨텍스트, 토큰 격리) |
| **8** | Critic / Verifier | Lead가 팀원 산출물 통합 검증 + 영역 침범/충돌 점검 + 환각 방지 검증 |
| **9** | Refiner | 충돌 시 Lead가 우선순위 결정 → 팀원에게 재작업 지시 |
| **10** | Output Renderer | 통합 산출물 + 팀원별 보고서 + 최종 사장 보고 |

### 단계별 검증/분기

| Step | 검증 | 실패 분기 |
|:-:|---|---|
| 1 | 요청 명확 + 팀 구성 필요성 정당화 | 단일 작업 → 일반 모드 fallback |
| 2 | CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1 환경 OK | 미충족 → 사장 알림 + 단일 에이전트 fallback |
| 3 | 팀원 1-6명 범위 | 7명+ → 분할 또는 우선순위 압축 |
| 4 | CLAUDE.md / 도메인 자료 / 제약 모두 공유 | 누락 → 보강 |
| 5 | 각 팀원 영역 명확 + 의존성 표 작성 | 모호 → 재분배 |
| 6 | 도구 실행 0 에러 (또는 부분 결과 OK) | 에러 → Lead가 재지시 또는 영역 재분배 |
| 7 | 모든 팀원 산출물 생성 + CLAUDE.md 규칙 준수 | 일부 누락 → 부분 통합 또는 보강 |
| 8 | 충돌/침범/환각 0건 | 발견 시 Step 9으로 |
| 9 | 충돌 해결 + 재작업 완료 | 미해결 → 사장 1차 판단 요청 |
| 10 | 통합 보고서 + 각 팀원 보고서 정합 | 모순 → 재정렬 |

### 인사이트 1 vs 본 스킬 — Confusion 방지

- 인사이트 1의 fast/normal/pro 모드 ↔ 본 스킬의 팀원 수(1-6명) 기반 — 다른 체계. 충돌 시 본 스킬 우선.
- 인사이트 1의 "한 모델 vs 다단계 호출" 원리 → 본 스킬은 **다중 에이전트 병렬 + Lead 통합** 으로 다단계 호출의 정점 형태.
- 본 스킬의 Lead/팀원 역할 분리 = 인사이트 1의 Step 8 Critic + Step 9 Refiner 분리와 정합.
