# CE Advisor — Context Engineering + Prompt Engineering Hybrid

## 사용하지 말아야 할 때 (NOT for)

- **즉시 실행 가능한 명확한 작업** — "제안해줘"/"suggest"/"advise"가 들어가도 의도가 구체적 산출물 생성(예: "이 버튼 색 제안해줘", "리팩토링 방법 제안해줘")이면 프롬프트 최적화가 아니라 그 작업을 바로 수행한다.
- **목적 자체가 불분명할 때** — 무엇을/왜 하려는지부터 정립이 필요하면 what / what-ce 사용.
- **단순 지식 질문 / 설명 요청** — 프롬프트를 다듬을 필요 없이 바로 답하면 된다.
- **코드·문서 등 실제 산출물 생성·수정 요청** — 본 스킬은 "프롬프트를 개선"하는 메타 작업 전용이며, 결과물 자체를 만들지 않는다.
- **이미 잘 정의된 짧은 단발 지시** — 최적화 오버헤드가 이득보다 큰 경우.

판단 기준: 사용자가 "내 프롬프트/요청문을 더 낫게 다듬어줘"라는 메타 의도를 가질 때만 활성화한다. 일상어 "제안/suggest/advise"가 작업 내용에 대한 제안 요청이면 비활성화한다.

> **원칙**: "가능한 가장 작은, 고신호(high-signal) 토큰 집합으로 원하는 결과의 확률을 극대화하라." — Anthropic

---

## 1. 트리거 조건

사용자가 `/ce-advisor`를 호출하거나, 복잡한 작업 요청 시 이 스킬을 적용한다.

**입력**: 사용자의 원본 프롬프트 또는 작업 의도
**출력**: 3+1개 최적화 제안 + 컨텍스트 진단

---

## 2. 실행 워크플로우 (2단계 인터랙티브)

### 1턴: 분석 (Analysis Turn)

Phase 0-2를 실행하고, 사용자에게 분석 결과를 보여준 뒤 확인을 받는다.

#### Phase 0: 컨텍스트 상태 스냅샷

```
📊 컨텍스트 스냅샷
├── 예상 사용량: [현재 대화 길이 기반 추정]
├── 여유도: [충분 / 보통 / 부족]
├── 활성 도구: [현재 사용 가능한 도구 수]
└── 권장 전략: [경량 / 표준 / 풍부]
```

**판단 기준**:
- 대화 초반 (첫 5턴): 여유 충분 → Context-Rich 권장
- 대화 중반 (5-15턴): 보통 → High-Signal 권장
- 대화 후반 (15턴+): 부족 → 최소 토큰 필수

#### Phase 1: 원본 프롬프트 분석 (7축)

| 축 | 분석 내용 |
|:--:|----------|
| **Goal** | 목표가 측정 가능한가? |
| **Output** | 산출물 형식이 구체적인가? |
| **Input** | 필요 데이터가 명시되었는가? |
| **Constraints** | 우선순위 포함 제약이 있는가? |
| **Eval** | 자가검증 기준이 있는가? |
| **Risks** | 4대 실패 모드 위험은? |
| **Context Arch** | 도구/메모리/RAG 설계가 적절한가? |

#### Phase 2: CE 실패 모드 경고

```
⚠️ CE 실패 모드 진단
├── Poisoning:   [🟢 안전 / 🟡 주의 / 🔴 위험] — [이유]
├── Distraction: [🟢 안전 / 🟡 주의 / 🔴 위험] — [이유]
├── Confusion:   [🟢 안전 / 🟡 주의 / 🔴 위험] — [이유]
└── Clash:       [🟢 안전 / 🟡 주의 / 🔴 위험] — [이유]
```

**진단 규칙**:
- **Poisoning**: 외부 데이터(RAG, MCP, 웹)를 검증 없이 사용하는가?
- **Distraction**: 요청에 현재 작업과 무관한 정보가 포함되었는가?
- **Confusion**: 도구/스킬이 겹치거나 모호한 결정 지점이 있는가?
- **Clash**: 지침 간 모순이 있는가? (글로벌 vs 프로젝트 vs 스킬)

#### 1턴 종료: AskUserQuestion으로 분석 확인

```
AskUserQuestion:
  header: "분석 확인"
  question: "분석 결과를 확인해주세요. 조정할 부분이 있나요?"
  options:
    1. "분석 OK, 제안 보여줘" — 2턴으로 진행
    2. "목표 수정" — Goal 축 재분석 필요
    3. "제약 추가" — Constraints 보강 필요
  multiSelect: false
```

---

### 2턴: 제안 + 선택 (Suggestion & Selection Turn)

1턴의 분석 + 사용자 피드백을 반영하여 3+1 제안을 생성한다.

#### Phase 3: 3+1 제안 생성

**제안 1: High-Signal (경량)**
- **원칙**: Right Altitude — 간결 쪽
- **토큰 목표**: 최소 (원본의 30-50%)
- **적용 기법**: 핵심 목표 + 출력 형식만. 배경/예시 제거.
- **적합 상황**: 대화 후반, 단순 작업, 컨텍스트 부족 시

```
[앞: 역할+목표 1줄] ━━ [뒤: 출력형식+제약]
```

**제안 2: Context-Rich (표준)**
- **원칙**: Right Altitude — 상세 쪽
- **토큰 목표**: 표준 (원본의 80-120%)
- **적용 기법**: 목표 + 배경 + 제약 + 예시 1-2개 + 검증 기준
- **적합 상황**: 대화 초반, 복잡한 작업, 컨텍스트 여유 시

```
[앞: 역할+목표] ━━ [중간: 배경+예시] ━━ [뒤: 출력형식+검증기준]
```

**제안 3: Multi-Turn (분할)**
- **원칙**: Isolate + Compress
- **토큰 목표**: 분산 (각 턴 200-400 토큰)
- **적용 기법**: 작업을 2-3단계로 분할, 각 단계별 프롬프트 제공
- **적합 상황**: 복잡한 분석, 멀티스텝 구현, 장기 작업

```
[Turn 1: 탐색/분석] → [Turn 2: 설계/계획] → [Turn 3: 구현/검증]
```

**제안 4: 함께 구체화 (협업)**
- **원칙**: 사용자와 대화를 통해 최적의 프롬프트를 정교화
- **진행 방식**:
  1. 원본 프롬프트의 모호한 부분 2-3개를 질문
  2. 사용자 답변을 반영하여 제안 1-3을 재생성
  3. 선택된 제안을 미세 조정

#### 2턴 종료: AskUserQuestion으로 제안 선택

```
AskUserQuestion:
  header: "CE 제안"
  question: "어떤 제안으로 실행할까요?"
  options:
    1. "제안 1: High-Signal" — 경량, 핵심만
    2. "제안 2: Context-Rich" — 표준, 상세
    3. "제안 3: Multi-Turn" — 분할 실행
    4. "제안 4: 함께 구체화" — 질문으로 정교화
  multiSelect: false
```

---

### 실행 (Execution)

선택 후 동작:
- **제안 1/2 선택** → 해당 프롬프트를 사용자 지시로 간주하여 **즉시 실행**
- **제안 3 선택** → Turn 1부터 순차 실행
- **제안 4 선택** → 질문 → 답변 반영 → 제안 재생성 → 2턴 반복
- **Other** → 사용자 커스텀 프롬프트로 실행

## 3. 토큰 예산 표시

모든 제안에 예상 토큰 비용을 표시한다:

```
💰 토큰 예산
├── 제안 1 (High-Signal):   ~[N] 토큰  [💚 경량]
├── 제안 2 (Context-Rich):  ~[N] 토큰  [💛 표준]
├── 제안 3 (Multi-Turn):    ~[N] 토큰  [💛 분산]
└── 제안 4 (함께 구체화):   ~[N]+ 토큰 [대화에 따라 가변]
```

---

## 4. 위치 최적화 시각화

각 제안에서 정보 배치를 시각적으로 보여준다:

```
📐 토큰 위치 배치
┌─────────────────────────────────────────────────┐
│ [PRIMACY] 역할, 목표, 핵심 제약                  │ ← 강한 주의력
├─────────────────────────────────────────────────┤
│ [MIDDLE] 배경 정보, 참고 데이터                  │ ← 주의력 저하 영역
├─────────────────────────────────────────────────┤
│ [RECENCY] 출력 형식, 검증 기준, Must Not         │ ← 강한 주의력
└─────────────────────────────────────────────────┘
```

**배치 규칙**:
- 역할/목표: 반드시 앞에
- 배경/데이터: 중간 (핵심이 아닌 보충 정보만)
- 출력 형식 + 검증 기준 + 절대 금지: 반드시 뒤에
- **핵심 정보는 절대 중간에 두지 않는다**

## 5. 이전 세션 패턴 학습

auto memory (`~/.claude/projects/*/memory/`)를 확인하여:

1. **이전 유사 작업**에서 어떤 제안 유형을 선택했는지 참조
2. **사용자 선호 패턴**이 있으면 기본 추천에 반영
3. 새로운 선택 패턴은 memory에 기록

```
🧠 패턴 학습
├── 이전 유사 작업: [있음/없음]
├── 선호 유형: [High-Signal / Context-Rich / Multi-Turn]
└── 이번 추천: 제안 [N]을 기본 추천
```

memory에 기록할 형식:
```
## CE Advisor 패턴
- [날짜] [작업유형] → 제안 [N] 선택 | 만족도: [높음/보통/낮음]
```

### 1턴 출력 (분석)

```markdown
# CE Advisor 분석

## 📊 컨텍스트 스냅샷
[Phase 0 결과]

## 🔍 프롬프트 분석 (7축)
[Phase 1 — 표]

## ⚠️ CE 실패 모드 진단
[Phase 2 결과]

→ [AskUserQuestion: 분석 확인]
```

### 2턴 출력 (제안)

```markdown
## 💡 제안

### 제안 1: High-Signal (~N 토큰) 💚
[실제 프롬프트]

### 제안 2: Context-Rich (~N 토큰) 💛
[실제 프롬프트]

### 제안 3: Multi-Turn (~N 토큰 x M턴) 💛
[각 턴별 프롬프트]

### 제안 4: 함께 구체화 🤝
[질문 목록]

## 💰 토큰 예산 비교
[표]

→ [AskUserQuestion: 제안 선택]
→ 선택 즉시 실행
```

## 7. 핵심 원칙 요약

| CE 원칙 | 이 스킬에서의 적용 |
|:-------:|-------------------|
| **Right Altitude** | 3개 제안이 간결↔상세 스펙트럼을 커버 |
| **Primacy/Recency** | 위치 최적화 시각화로 배치 가이드 |
| **Compress** | High-Signal 제안이 압축 전략 제시 |
| **Select** | Context-Rich가 필요 정보만 선별 |
| **Isolate** | Multi-Turn이 작업별 컨텍스트 분리 |
| **4대 실패 모드** | 매 분석마다 자동 진단 |

## 8. 자기 제약 (Anti-Bloat)

이 스킬 자체가 CE 원칙을 위반하지 않도록:

- 출력은 **500토큰 이내**를 목표로 한다 (제안 프롬프트 제외)
- 불필요한 반복 설명을 하지 않는다
- 사용자가 제안을 선택하면 **분석/제안 출력을 종료하고, 선택된 프롬프트를 사용자 지시로 간주하여 즉시 실행**한다. 추가 확인 없이 바로 작업 착수.
- 매번 모든 섹션을 출력하지 않고, **핵심 변화가 있는 섹션만** 표시한다

## 9. Confirmation Loop

> **프로토콜 참조**: `_core/confirmation-loop.md`

이 스킬은 **2턴 인터랙티브** 구조에서 각 턴 종료 시 확인 루프를 적용한다.

**1턴 (분석 확인) 훅 오버라이드:**

| 훅 | 값 |
|:--:|-----|
| `PRESENT_FORMAT` | CE 7축 분석표 + 실패 모드 진단 + 컨텍스트 스냅샷 |
| `CONFIRM_OPTIONS` | ["분석 OK, 제안 보여줘", "목표 수정", "제약 추가"] |
| `MODIFY_OPTIONS_BUILDER` | 7축 중 수정 대상 축 선택지 |
| `STEP_LABEL` | "CE 분석 (Phase 0-2)" |

**2턴 (제안 선택) 훅 오버라이드:**

| 훅 | 값 |
|:--:|-----|
| `PRESENT_FORMAT` | 3+1 제안 카드 + 토큰 예산표 + 위치 배치도 |
| `CONFIRM_OPTIONS` | ["제안 1: High-Signal", "제안 2: Context-Rich", "제안 3: Multi-Turn", "제안 4: 함께 구체화"] |
| `MODIFY_OPTIONS_BUILDER` | 제안별 세부 항목 수정 선택지 |
| `STEP_LABEL` | "3+1 제안 검토" |
