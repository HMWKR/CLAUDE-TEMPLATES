# First Principles Thinking — 전제 해체 + 재구성

> **핵심 원리** (Aristotle / Elon Musk): "유추(analogy)가 아닌 기본 진실(fundamental truths)에서 추론하라."
> 더 이상 분해할 수 없는 기본 전제에 도달한 후, 그 전제에서 새롭게 재구성한다.

**Iron Law**: 전제 검증 결과를 사용자에게 제시하고 확인받은 후에만 재구성을 실행한다.

## Announce Pattern

> "First Principles로 현재 접근법의 전제를 해체합니다.
> 기존 가정을 나열 → 분해(Why Chain) → 전제 검증 → 재구성 4단계로 진행하며,
> 각 단계마다 확인을 받겠습니다."

---

## Trigger Rules

### 트리거 (작동)

- `/first-principles` (슬래시 커맨드)
- "근본부터 다시 생각해", "왜 이렇게 하는 건지 의문이야"
- "전제를 의심해봐", "당연하다고 생각하는 게 맞나"
- "기존 방식 말고 새로운 접근", "처음부터 다시 생각해보자"
- "관행을 따르는 게 맞아?", "이 결정의 근거가 뭐야"
- 아키텍처 결정, 기술 선택 등에서 "왜?"가 답이 안 될 때

### 비트리거 (작동 안 함)

- 이미 명확한 해결책이 있는 경우
- 긴급 상황 (→ Cynefin Chaotic)
- 목적 정립이 필요한 경우 (→ `/what`)
- "First Principles"가 텍스트 내용으로 언급된 경우

## 실행 플로우

### STEP 1: 현재 접근법/가정 나열

현재 문제를 해결하기 위해 **암묵적으로 전제하고 있는 것들**을 도출한다.

방법:
1. 현재 결정/접근법을 1문장으로 정리
2. "이 결정은 무엇을 전제하는가?" 질문
3. 5-10개 가정을 도출 (명시적 + 암묵적)

출력 형식:
```
## 현재 접근법
> "[현재 결정/접근법 1문장]"

## 전제 목록
| # | 가정 | 유형 |
|---|------|------|
| 1 | [가정 내용] | 명시적 / 암묵적 |
| 2 | ... | ... |
```

유형 구분:
- **명시적**: 문서/코드/대화에서 직접 언급된 전제
- **암묵적**: 언급되지 않았지만 결정의 기저에 있는 전제

→ Confirmation Loop ("이 가정들이 맞나요? 빠진 것이 있나요?")

### STEP 2: 가정 분해 (Why Chain)

각 가정에 "이것이 진실인 이유는?" 을 반복하여 **기본 진실(axiom)**에 도달한다.

분해 깊이:
- 단순 가정: 2-3단계
- 복잡 가정: 4-5단계
- 더 이상 "왜?"에 답할 수 없으면 → 기본 진실

분해 트리 형식:
```
가정 1: "[가정 내용]"
  ↓ 왜?
  이유 1-1: "[이유]"
    ↓ 왜?
    이유 1-1-1: "[더 깊은 이유]"
      ↓ 왜?
      ★ 기본 진실: "[더 이상 분해 불가한 사실]"
```

모든 가정에 대해 분해 트리를 완성한 후, 기본 진실 목록 정리:

```
## 기본 진실 (Axioms)
| # | 기본 진실 | 출처 가정 |
|---|----------|----------|
| A | [분해 불가한 사실] | 가정 1, 3 |
| B | [분해 불가한 사실] | 가정 2 |
```

→ Confirmation Loop ("분해 결과가 맞나요?")

### STEP 3: 전제 검증

각 기본 진실과 중간 가정의 **현재 유효성**을 검증한다.

검증 방법:
| 방법 | 적용 대상 |
|------|----------|
| 코드/문서 확인 | 기술적 전제 (Read, Grep으로 실제 확인) |
| 도메인 지식 | 비즈니스/시장 전제 |
| 시간 검증 | "이 전제가 설정된 시점 vs 현재" |
| 반례 탐색 | "이 전제가 틀린 경우가 있는가?" |

검증 결과 마커:
- **[유효]**: 현재도 참. 유지.
- **[무효]**: 현재 거짓이거나 상황 변경. **핵심 발견**.
- **[조건부]**: 특정 조건에서만 참. 조건 명시.
- **[미검증]**: 확인 불가. 리스크로 기록.

출력 형식:
```
## 전제 검증 결과
| # | 가정 | 기본 진실 | 검증 | 상태 |
|---|------|----------|------|------|
| 1 | [가정] | [axiom] | [검증 방법+결과] | [유효/무효/조건부/미검증] |

## 핵심 발견: 무효화된 전제
| # | 무효 전제 | 무효 이유 | 영향 범위 |
|---|----------|----------|----------|
```

→ Confirmation Loop ("검증 결과가 맞나요? 특히 무효 판정에 동의하시나요?")

### STEP 4: 재구성

유효한 전제만으로 새로운 접근법을 구성한다.

재구성 프로세스:
1. 무효화된 전제를 제거
2. 유효한 기본 진실만으로 새 접근법 도출
3. 기존 접근법과 비교

출력 형식:
```
## 재구성

### 기존 접근법 (유추 기반)
> "[기존 결정]" — 전제 [1, 2, 3] 포함 (전제 2 무효)

### 재구성 접근법 (전제 기반)
> "[새 결정]" — 유효 전제 [1, 3, A, B]만 사용

### 비교표
| 항목 | 기존 | 재구성 |
|------|------|--------|
| 기반 전제 수 | [N]개 (무효 [M]개 포함) | [K]개 (전부 유효) |
| [비교 축 1] | ... | ... |
| [비교 축 2] | ... | ... |
```

→ Confirmation Loop ("재구성 접근법을 채택할까요?")

### STEP 5: 실행 또는 Handoff

- **즉시 실행**: 재구성된 접근법으로 직접 작업 시작
- **what Handoff**: 재구성된 접근법을 what 스킬의 What/How에 주입
- **architect Handoff**: 새 접근법으로 아키텍처 재설계
- **기존 유지**: 사용자가 기존 접근법 유지를 선택

| 스킬 | 연동 지점 |

## CE 기여

| CE 실패 모드 | First Principles의 방어 |
|:------------:|:-----------------------|
| **Confusion** | 상충하는 지시의 근본 전제를 드러내 모순 해소 |
| **Poisoning** | "이전 프로젝트에서 이렇게 했으니까"라는 유추 기반 오염 차단 |
| **적정 고도** | 전제를 분해함으로써 적절한 추상화 수준 발견 |

## Edge Cases

| 상황 | 처리 |
|------|------|
| 모든 전제가 유효 | "현재 접근법의 전제가 건전합니다" → 기존 계획 유지 권고 |
| 모든 전제가 무효 | 근본적 재설계 필요 → what 스킬로 목적부터 재정립 |
| 사용자가 "너무 깊다" | 분해 깊이를 2단계로 제한하고 핵심 전제만 검증 |
| 전제가 30개 이상 | 영향도 높은 상위 5-7개만 선별하여 분석 |
| 시간 제약 | STEP 1-3을 한 번에 제시하고 일괄 확인 (Fast Mode) |

## Red Flags

**Never:**
- 검증 없이 전제를 "무효"로 선언하지 않는다 (반드시 근거 제시)
- 유추를 사용하여 재구성하지 않는다 (전제에서만 도출)
- 사용자의 도메인 지식을 무시하고 전제를 판단하지 않는다
- 분해를 무한히 계속하지 않는다 (기본 진실 도달 시 중단)
