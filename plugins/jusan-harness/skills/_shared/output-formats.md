# 공통 출력 형식 — Output Format Templates

> 여러 스킬에서 공통으로 사용하는 출력 형식 템플릿.
> 스킬별 SKILL.md에서 `~/.claude/skills/_shared/output-formats.md` 경로로 참조.

---

## 1. 분석 보고서 형식

분석/감사/리뷰 계열 스킬의 표준 보고서 구조.

```markdown
# [분석 유형] 보고서

## 요약
| 항목 | 값 |
|------|-----|
| 대상 | [분석 대상] |
| 일시 | [실행 일시] |
| 점수 | [N/100] |
| 등급 | [S/A/B/C/F] |

## 핵심 발견사항
1. [Critical] ...
2. [Major] ...
3. [Minor] ...

## 상세 분석
### [영역 1]
- 현상: ...
- 원인: ...
- 권장 조치: ...

## 권장 조치 (우선순위순)
1. [즉시] ...
2. [단기] ...
3. [중기] ...
```

### 사용 스킬
- `infra-audit`: 인프라 감사 보고서
- `security-audit`: 보안 감사 보고서
- `playwright-uiux-audit`: UI/UX 감사 보고서
- `playwright-design-audit`: 디자인 감사 보고서
- `domain-expert-analysis`: 도메인 전문가 분석

## 2. 점수 매트릭스 형식

정량 평가가 필요한 스킬의 점수 표 구조.

```markdown
## 점수 매트릭스

| # | 영역 | 점수 | 가중치 | 가중 점수 | 상태 |
|:-:|------|:----:|:-----:|:--------:|:----:|
| 1 | [영역1] | N | N% | N×가중치 | PASS/WARN/FAIL |
| 2 | [영역2] | N | N% | N×가중치 | PASS/WARN/FAIL |
| | **합계** | | **100%** | **N** | |
```

### 등급 체계 (공통)

| 점수 범위 | 등급 | 의미 |
|:---------:|:----:|------|
| 95-100 | S | 탁월 |
| 85-94 | A | 우수 |
| 70-84 | B | 양호 |
| 50-69 | C | 개선 필요 |
| 0-49 | F | 긴급 조치 |

### 사용 스킬
- `infra-audit`: 7영역 42항목 점수
- `playwright-uiux-audit`: 18차원 UX 점수
- `playwright-design-audit`: 19전문가 디자인 점수

## 3. 이슈 목록 형식

QA/테스트 계열 스킬의 이슈 리포트 구조.

```markdown
## 발견된 이슈

### [ISS-001] 이슈 제목
- **심각도**: Critical / Major / Minor / Info
- **카테고리**: [기능/접근성/성능/보안/UX]
- **위치**: [파일:라인 또는 URL + 선택자]
- **현상**: [무엇이 잘못되었는지]
- **기대 동작**: [어떻게 동작해야 하는지]
- **재현 경로**: [단계별 재현 방법]
- **권장 수정**: [어떻게 고쳐야 하는지]
- **참고**: [관련 표준/가이드라인]
```

### 심각도 기준

| 심각도 | 기준 | 대응 시한 |
|:------:|------|:---------:|
| Critical | 핵심 기능 불가, 데이터 손실, 보안 취약점 | 즉시 |
| Major | 주요 기능 저하, 접근성 위반, 성능 심각 | 1주 내 |
| Minor | 사소한 UI 결함, 개선 가능 영역 | 다음 릴리스 |
| Info | 참고 사항, 모범 사례 권장 | 선택적 |

### 사용 스킬
- `playwright-qa-expert`: QA 테스트 이슈
- `playwright-qa-agent-teams`: 팀 QA 이슈
- `continuous-qa-loop`: 반복 QA 이슈
- `security-audit`: 보안 이슈

## 4. 코드 리뷰 형식

코드 리뷰/리팩토링 스킬의 피드백 구조.

```markdown
## 코드 리뷰 결과

### 요약
- 검토 파일: N개
- 발견 사항: Critical N / Major N / Minor N / Suggestion N

### 발견 사항

#### [CR-001] [카테고리] 제목
- **파일**: `path/to/file.ts:42`
- **심각도**: Critical / Major / Minor / Suggestion
- **설명**: [무엇이 문제인지]
- **수정 제안**:
  ```diff
  - 기존 코드
  + 수정 코드
  ```
- **근거**: [왜 변경이 필요한지]
```

### 리뷰 카테고리

| 카테고리 | 설명 |
|----------|------|
| Security | 보안 취약점 (OWASP Top 10) |
| Architecture | 설계 패턴, 의존성, 결합도 |
| Performance | 성능 병목, 메모리 누수 |
| Testing | 테스트 커버리지, 엣지 케이스 |
| Readability | 가독성, 네이밍, 문서화 |
| Best Practice | 언어/프레임워크 모범 사례 |

### 사용 스킬
- `agent-teams-code-review`: 팀 코드 리뷰
- `js-refactor-cleanup-skill`: JS/TS 리팩토링
- `unused-code-refactor-suggester`: 미사용 코드 분석

## 5. 의사결정 프레임워크 형식

분석/자문 스킬의 의사결정 지원 구조.

```markdown
## 의사결정 분석

### 문제 정의
[해결해야 할 핵심 문제 1문장]

### 옵션 비교

| 기준 | 옵션 A | 옵션 B | 옵션 C |
|------|:------:|:------:|:------:|
| [기준1] | [평가] | [평가] | [평가] |
| [기준2] | [평가] | [평가] | [평가] |
| **종합** | **N/10** | **N/10** | **N/10** |

### 추천
**옵션 [X]** 추천. 근거: [1-2문장]

### 리스크 및 완화 방안
1. [리스크]: [완화 방안]
```

### 사용 스킬
- `ce-advisor`: CE+PE 프롬프트 최적화 제안
- `what`: Why-What-How-So What 분석
- `deep-analysis-mode`: 심층 분석 의사결정
- `architect`: 아키텍처 설계 의사결정

## 6. 진행 상태 보고 형식

멀티에이전트/장기 작업 스킬의 진행 보고 구조.

```markdown
## 진행 상태

| 단계 | 상태 | 담당 | 비고 |
|------|:----:|------|------|
| 1. [단계명] | DONE | [에이전트] | [결과 요약] |
| 2. [단계명] | IN_PROGRESS | [에이전트] | [현재 진행] |
| 3. [단계명] | PENDING | - | [대기 조건] |

### 전체 진행률: N/M (N%)
```

### 사용 스킬
- `agent-teams-feature-dev`: 기능 개발 파이프라인
- `continuous-qa-loop`: 반복 QA 루프
- `agent-teams-orchestrator`: 팀 오케스트레이션

---

## 참조 규칙

1. **형식 선택**: 스킬 성격에 맞는 형식 선택. 복수 형식 조합 가능.
2. **일관성**: 동일 스킬 내에서는 항상 같은 형식 사용.
3. **확장**: 스킬별 특화 필드 추가 가능하되, 기본 구조는 유지.
4. **언어**: 한국어 기본. 기술 용어는 원어 병기 가능.
