# Project Bootstrapper — 웹/앱 프로젝트 부트스트래핑 파이프라인

## 핵심 철학

사용자가 "OO 사이트 만들어줘"라고 한마디 하면, **에이전트 팀 구성부터 프로토타입 코드까지** 체계적으로 설계하여 즉시 개발에 착수할 수 있는 파일 패키지를 제공한다. 이 스킬은 PetFit AI 쇼핑몰 프로젝트에서 검증된 파이프라인을 범용화한 것이다.

---

## 실행 전 사전 작업

### 1단계: 프로젝트 의도 파악

사용자의 요청에서 다음 정보를 추출하거나 질문한다:

- **서비스 유형**: 쇼핑몰, SaaS, 커뮤니티, 마켓플레이스, 교육, 예약 등
- **핵심 기능**: 가장 중요한 1~3개 기능 (예: "AI 피팅", "실시간 채팅", "결제")
- **타겟 사용자**: 누가 쓰는가 (연령, 직업, 행동 패턴)
- **기술 선호**: 특정 기술 스택 요구가 있는가 (없으면 프로젝트에 최적인 것을 추천)
- **부가 요청**: 에이전트 팀 규모, CLI 환경 최적화, 사업계획서 필요 여부

정보가 부족하면 질문한다. 단, 사용자가 "알아서 해줘" 류의 반응이면 합리적으로 추론하여 진행한다.

### 2단계: 실행 모드 선택

사용자에게 두 가지 모드를 제시한다:

- **자동 모드**: 7단계를 한 번에 실행하고 최종 ZIP으로 제공. 빠르지만 중간 수정 불가.
- **단계별 모드**: 각 단계마다 산출물을 보여주고 확인을 받은 뒤 다음 단계로 진행.

사용자가 선택하지 않으면 **자동 모드**를 기본으로 한다.

---

## 7단계 파이프라인

각 단계의 상세 템플릿과 규칙은 `references/` 디렉토리의 파일을 읽어서 적용한다. SKILL.md에는 각 단계의 목적과 산출물만 정의하고, 구체적인 생성 규칙은 references에 위임한다.

### STAGE 1: 에이전트 팀 구성

**목적**: 프로젝트 도메인에 맞는 전문가 에이전트 팀을 자동 생성한다.

**산출물**: `agents/{부서}/team.md` (8개 부서, 각 6명 이상)

**규칙**:
- `references/agent-templates.md`를 읽고 따른다.
- 8개 부서는 고정: PM/기획, AI/ML(해당 시), 프론트엔드, 백엔드, UX/디자인, 데이터, QA/DevOps, 비즈니스
- 프로젝트에 AI 기능이 없으면 AI/ML 부서 대신 해당 도메인 전문 부서로 대체 (예: 교육 → 콘텐츠 기획팀)
- 각 에이전트에 반드시 포함: 이름(한국어), 직책, 전문 분야, 행동 규칙, CLI용 프롬프트
- 의사결정 우선순위 위계를 정의: PM > CTO > 부서 리드 > 개별

### STAGE 2: CLAUDE.md 시스템 프롬프트

**목적**: Claude Code CLI에 바로 적용할 수 있는 마스터 프롬프트를 생성한다.

**산출물**: `CLAUDE.md`

**규칙**:
- `references/claudemd-template.md`를 읽고 따른다.
- 반드시 포함할 섹션: 프로젝트 개요, 핵심 기능, 기술 스택, 코딩 컨벤션, 아키텍처 원칙, 에이전트 팀 구조, 에이전트 호출 방법, 금지사항
- 기술 스택은 프로젝트 유형에 따라 자동 선택하되 사용자가 지정했으면 그것을 따른다
- 금지사항은 프로젝트별 핵심 리스크에서 도출

### STAGE 3: 기술 아키텍처 + 다이어그램

**목적**: 시스템 아키텍처를 설계하고 시각적 다이어그램을 HTML 파일로 제공한다.

**산출물**: 
- `docs/architecture.md` (텍스트 설계서)
- `diagrams/01-system-architecture.html` (인프라 구성도 SVG)
- `diagrams/02-data-flow.html` (데이터 흐름 SVG)
- 프로젝트에 따라 추가 다이어그램 (AI 파이프라인 등)

**규칙**:
- `references/diagram-templates.md`를 읽고 따른다.
- 다이어그램은 브라우저에서 바로 열 수 있는 standalone HTML (SVG 내장)
- 다크 테마 기본, 컬러는 기능 카테고리별로 구분
- 각 노드에 지연시간/비용 등 정량적 정보를 라벨로 표시

### STAGE 4: MVP 와이어프레임 + UX 플로우

**목적**: 핵심 화면의 인터랙티브 와이어프레임과 유저 저니 플로우를 제공한다.

**산출물**:
- `docs/mvp-wireframe-spec.md` (화면별 상세 스펙)
- `wireframes/00-user-journey-flow.html` (전체 유저 저니 SVG)
- `wireframes/01-XX.html` ~ (각 화면 와이어프레임 HTML)
- `wireframes/error-screens.html` (에러 화면 모음)

**규칙**:
- `references/wireframe-templates.md`를 읽고 따른다.
- 모든 와이어프레임은 모바일 퍼스트 (340px 폰 프레임)
- 인터랙티브: 단계별 스테퍼, 탭 전환, 슬라이더 등 동작하는 HTML
- 에러 화면은 반드시 포함 (최소 5종: 입력 불량, 인식 실패, 처리 실패, 네트워크, 서버)
- 모든 에러 메시지에 해결 방법 포함, 기술 용어 금지
- 유저 저니 플로우에 전환율 목표를 명시

### STAGE 5: DB 스키마 + API 설계

**목적**: 데이터베이스 스키마와 API 엔드포인트를 설계한다.

**산출물**:
- `db/schema.sql` (PostgreSQL DDL)
- `docs/api-spec/api-endpoints.md` (REST API 명세)

**규칙**:
- `references/db-api-templates.md`를 읽고 따른다.
- DB: PostgreSQL 기준, 인덱스 포함, 핵심 쿼리 예시를 주석으로
- API: RESTful, 응답 형식 표준화 `{ success, data, error }`, 에러 코드 테이블 포함
- 인증, 핵심 CRUD, 비즈니스 로직, 실시간(WebSocket 해당 시)을 모두 커버
- 프로젝트의 핵심 데이터 루프(피드백, 추천 등)를 지원하는 테이블을 반드시 포함

### STAGE 6: 프로토타입 코드

**목적**: 즉시 개발에 착수할 수 있는 프론트엔드 + 상태관리 코드를 제공한다.

**산출물**:
- `src/types/index.ts` (전체 타입 정의)
- `src/lib/` (도메인 데이터, 유틸리티 함수, 목업 데이터)
- `src/store/` (Zustand 스토어 — 핵심 도메인별)
- `src/app/page.tsx` 또는 동등한 메인 컴포넌트
- `package.json` + `tailwind.config.ts`
- 인터랙티브 프로토타입 `.jsx` (브라우저에서 바로 확인 가능)

**규칙**:
- `references/code-templates.md`를 읽고 따른다.
- 기본 스택: Next.js 14+ (App Router) + Tailwind CSS + Zustand (사용자가 다른 스택을 요청하면 변경)
- 타입 정의는 DB 스키마와 일치해야 함
- 목업 데이터로 동작하는 프로토타입 (실제 백엔드 없이 UI 확인 가능)
- 프로토타입 JSX는 전체 유저 저니를 한 파일로 — 온보딩부터 완료까지

### STAGE 7: 사업계획서 + 인프라 설계

**목적**: 비즈니스 관점의 사업계획서와 인프라 비용 추정을 제공한다.

**산출물**:
- `docs/business-plan/executive-summary.md`
- `infra/aws-architecture.md` (또는 해당 클라우드)
- `CLI-SETUP-GUIDE.md` (전체 패키지 사용 가이드)

**규칙**:
- `references/business-templates.md`를 읽고 따른다.
- 사업계획서: 문제정의, 솔루션, 시장, 비즈니스모델, 경쟁우위, 재무계획, 리스크
- 수치(시장 규모, 비용 등)에 반드시 [추정] 또는 [검색확인] 태그 — 웹 검색으로 확인 가능한 것은 검색
- 인프라: AWS 기본, 서비스별 구성 + MAU별 비용 추정표
- CLI 가이드: 패키지 적용 방법 + 실전 프롬프트 예시 20개+

## 최종 패키징

모든 단계 완료 후:

1. 전체 디렉토리를 ZIP으로 압축
2. 워크스페이스에 파일로 산출 (ZIP 및 산출물을 작업 디렉토리에 저장)
3. 최종 요약을 제시: 파일 수, 에이전트 수, 핵심 산출물 체크리스트

**디렉토리 구조 표준**:
```
{project-name}/
├── CLAUDE.md
├── CLI-SETUP-GUIDE.md
├── package.json
├── tailwind.config.ts
├── agents/{부서}/team.md (8개)
├── docs/
│   ├── domain-knowledge/ (도메인별 날리지)
│   ├── api-spec/api-endpoints.md
│   ├── business-plan/executive-summary.md
│   └── mvp-wireframe-spec.md
├── workflows/ (태스크 위임, 코드 리뷰, 배포)
├── diagrams/ (아키텍처 SVG HTML)
├── wireframes/ (인터랙티브 와이어프레임 HTML)
├── db/schema.sql
├── infra/aws-architecture.md
└── src/
    ├── types/index.ts
    ├── lib/ (데이터, 유틸)
    ├── store/ (Zustand)
    └── app/ (메인 컴포넌트)
```

## 자기 검증 (모든 산출물에 적용)

각 단계의 산출물을 제출하기 전에 다음 3가지를 검증한다:

1. **(a) 완전성**: 빠뜨린 관점이 있는가? 에러 케이스는? 엣지 케이스는?
2. **(b) 품질**: 업계 표준과 비교했을 때 수준이 충분한가?
3. **(c) 비판적 검증**: 이 프로젝트가 실패할 수 있는 이유 3가지는?

검증 결과를 산출물과 함께 별도 섹션으로 명시한다.

## 도메인 적응 규칙

프로젝트 유형별로 파이프라인을 자동 조정한다:

| 프로젝트 유형 | 특수 에이전트 | 핵심 기술 | 추가 다이어그램 |
|-------------|------------|----------|-------------|
| 이커머스/쇼핑몰 | MD/바잉, 물류 | 결제PG, 상품관리 | 주문 플로우 |
| SaaS | 요금제 설계 | 구독 결제, 멀티테넌시 | 요금제 비교 |
| 커뮤니티 | 모더레이션 | 실시간(WS), 피드 | 콘텐츠 플로우 |
| 교육 플랫폼 | 콘텐츠 기획, 강사 | LMS, 영상스트리밍 | 학습 플로우 |
| 예약 시스템 | 스케줄링 | 캘린더, 알림 | 예약 상태머신 |
| AI 서비스 | AI/ML 전문 (확대) | GPU 서빙, 큐 | AI 파이프라인 |
| 마켓플레이스 | 판매자/구매자 양면 | 에스크로, 리뷰 | 양면 플로우 |

해당 유형에 해당하지 않으면 가장 가까운 유형을 기준으로 적응한다.

## 주의사항

- 이 스킬은 **프로젝트 "전체"를 부트스트래핑**할 때 사용한다. 컴포넌트 하나, 함수 하나를 요청하는 경우에는 사용하지 않는다.
- 사업계획서의 시장 규모, 비용 등 수치는 반드시 [추정] 태그를 붙이고, 가능하면 웹 검색으로 교차검증한다.
- 에이전트 이름은 한국어 실명 스타일로 생성한다 (실제 인물이 아닌 가상).
- 모든 사용자 대면 텍스트(에러 메시지, UX 카피)는 한국어로 작성한다.
- 와이어프레임 HTML은 standalone으로 — 외부 CDN 의존 없이 브라우저에서 바로 열 수 있어야 한다.
