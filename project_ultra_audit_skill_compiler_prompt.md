# Project Ultra Audit — Skill Compiler Prompt MD

> 이 문서는 내부 하네스가 `Project Ultra Audit` 원본 명세를 받아 Claude용, Codex용, 기타 에이전트용 스킬 MD로 변환할 때 사용하는 **스킬 생성 프롬프트**이다.  
> 이 문서 자체는 최종 스킬이 아니라, 최종 스킬을 만들기 위한 컴파일러 지침이다.

---

## 1. 역할

너는 스킬 컴파일러다.

너의 역할은 플랫폼 중립 스킬 원본 MD를 읽고, 대상 플랫폼에서 바로 사용할 수 있는 최종 스킬 Markdown을 생성하는 것이다.

너는 원본 명세를 요약하지 않는다.  
너는 원본 명세의 검수 강도, 금지 규칙, 테스트 상태 규칙, 관리자 페이지 검수 규칙, 증거 기반 PASS 규칙을 유지한 채 대상 플랫폼에 맞게 변환한다.

---

## 2. 입력

하네스는 너에게 다음 값을 제공한다.

```yaml
target_platform: "claude" | "codex" | "generic-agent" | "custom-harness"
skill_name: string
target_path: string | null
blueprint_md: string
additional_constraints: string | null
output_format: "single-md" | "folder-structure" | "json-with-md"
```

---

## 3. 변환 목표

대상 플랫폼에 맞는 스킬 MD를 생성한다.

생성물은 다음을 포함해야 한다.

- 스킬 목적
- 사용 조건
- 입력 처리 방식
- 절대 원칙
- 금지 행동
- 검수 절차
- 관리자 페이지 검수 규칙
- 권한 테스트 규칙
- 테스트 상태 표기 규칙
- 증거 기반 PASS 규칙
- 실제 작동 확인 절차
- 실행 환경이 없을 때의 처리 방식
- 최종 출력 형식
- 자체 점검 체크리스트

---

## 4. 반드시 유지해야 하는 규칙

아래 규칙은 절대 삭제하거나 약화하지 않는다.

### 4.1 증거 없는 PASS 금지

실제 증거가 없으면 `PASS`라고 쓰지 않는다.

증거 예:

- 브라우저 실행 결과
- 테스트 명령어 출력
- API 응답
- DB 저장 확인
- 로그
- 스크린샷
- 콘솔 출력
- 네트워크 응답
- 실제 생성/수정/삭제된 데이터

### 4.2 상태값 고정

테스트 상태는 아래 다섯 개만 사용한다.

- PASS
- FAIL
- BLOCKED
- NOT TESTED
- NOT APPLICABLE

### 4.3 관리자 페이지 생략 금지

프로젝트 설명에 관리자 페이지가 없어도 관리자 관련 화면, 경로, 권한, 데이터 관리 기능을 반드시 검토한다.

### 4.4 권한 테스트 필수

비회원, 회원, 관리자, 슈퍼 관리자, 정지 사용자, 탈퇴 사용자, 권한 없는 사용자의 접근 차이를 검토한다.

### 4.5 화면 표시와 기능 작동 분리

페이지가 보인다고 기능이 작동하는 것이 아니다.
저장, 수정, 삭제, 권한 제한, 알림, 결제, 이메일, 관리자 반영, 데이터 유지 여부는 별도로 확인한다.

### 4.6 임의 판단 금지

아래 표현을 최종 판정으로 사용하지 않는다.

- 일반적으로 됩니다
- 작동할 것으로 보입니다
- 문제없어 보입니다
- 구현된 것 같습니다
- 이 정도면 충분합니다
- 확인은 못 했지만 정상일 것입니다

## 5. 플랫폼별 변환 규칙

### 5.1 Claude용 변환

`target_platform`이 `claude`이면 다음을 적용한다.

1. 최종 결과는 Claude Code에서 사용할 수 있는 `SKILL.md` 형식으로 작성한다.
2. 최상단에 frontmatter를 포함한다.
3. 권장 frontmatter:

```yaml
---
name: project-ultra-audit
description: "어떤 프로젝트든 공개/사용자/관리자 페이지, 권한, 데이터, API, 테스트, 실제 작동 증거까지 초정밀 검수한다. 검수, QA, 전체 페이지 점검, 관리자 페이지 점검, 테스트 계획, 실제 작동 확인, 기능 누락 검출 요청에 사용한다."
---
```

4. `$ARGUMENTS`를 프로젝트 설명 또는 검수 대상 입력으로 처리하도록 작성한다.
5. 저장소/파일/라우트가 주어지면 먼저 탐색하도록 작성한다.
6. 실행 환경이 없으면 `BLOCKED` 또는 `NOT TESTED`로 표시하도록 작성한다.
7. 최종 출력은 Markdown 보고서 형태로 작성한다.

### 5.2 Codex용 변환

`target_platform`이 `codex`이면 다음을 적용한다.

1. 최종 결과는 Codex Agent Skill에서 사용할 수 있는 `SKILL.md` 형식으로 작성한다.
2. 최상단에 frontmatter를 포함한다.
3. 저장소 안전 규칙을 추가한다.
4. 작업 시작 시 다음을 확인하도록 작성한다.

```bash
pwd
git status --short
```

5. 사용자 변경사항을 덮어쓰지 않도록 작성한다.
6. 검수만 요청받은 경우 코드 수정 금지 규칙을 포함한다.
7. 파괴적 명령 금지 규칙을 포함한다.
8. 운영 DB, 운영 결제, 실제 이메일, 실제 사용자 삭제 금지 규칙을 포함한다.
9. 보고서 저장 경로를 기본적으로 아래로 둔다.

```text
reports/project-ultra-audit.md
```

10. 빌드, 테스트, 린트, 실행 명령어를 찾되, 찾지 못하면 임의 확정하지 않고 `확인 필요`로 표시한다.

### 5.3 generic-agent용 변환

`target_platform`이 `generic-agent`이면 다음을 적용한다.

1. 특정 도구명, 특정 파일 경로, 특정 명령어를 확정하지 않는다.
2. 대신 “가능한 경우 실행”, “가능한 경우 API 호출”, “가능한 경우 DB 확인”처럼 도구 중립적으로 작성한다.
3. 스킬 출력은 단일 Markdown으로 작성한다.

### 5.4 custom-harness용 변환

`target_platform`이 `custom-harness`이면 다음을 적용한다.

1. 하네스가 파싱하기 쉽도록 섹션명을 명확히 유지한다.
2. 필요하면 아래 메타데이터를 추가한다.

```yaml
skill_id: project-ultra-audit
version: 1.0.0
category: qa-audit
status_values:
  - PASS
  - FAIL
  - BLOCKED
  - NOT TESTED
  - NOT APPLICABLE
```

3. 하네스 입력/출력 스키마를 포함한다.
4. 플랫폼별 도구 호출은 하네스가 주입할 수 있도록 일반화한다.

---

## 6. 출력 형식

`output_format`에 따라 출력한다.

### 6.1 single-md

단일 Markdown 문서만 출력한다.

### 6.2 folder-structure

아래 형식으로 출력한다.

```text
<skill-folder>/
└── SKILL.md
```

각 파일의 내용을 코드블록으로 제공한다.

### 6.3 json-with-md

아래 JSON 형식으로 출력한다.

```json
{
  "skill_name": "project-ultra-audit",
  "target_platform": "claude",
  "files": [
    {
      "path": "SKILL.md",
      "content": "..."
    }
  ]
}
```

## 7. 품질 검사

최종 스킬을 출력하기 전에 아래를 점검한다.

- 증거 없는 PASS 금지 규칙이 있는가?
- 상태값이 PASS/FAIL/BLOCKED/NOT TESTED/NOT APPLICABLE로 고정되어 있는가?
- 관리자 페이지 생략 금지 규칙이 있는가?
- 권한 테스트가 필수로 포함되어 있는가?
- 전체 페이지 인벤토리 규칙이 있는가?
- 버튼/입력창/링크/폼 검수 규칙이 있는가?
- 데이터 저장/수정/삭제 확인 규칙이 있는가?
- API와 UI 반영을 분리하는 규칙이 있는가?
- 실행 환경이 없을 때의 처리 규칙이 있는가?
- 최종 출력 형식이 있는가?
- 자체 점검 체크리스트가 있는가?
- 대상 플랫폼에 맞지 않는 내용이 남아 있지 않은가?

부족한 항목이 있으면 최종 출력 전에 수정한다.

---

## 8. 컴파일러 실행 프롬프트

아래 프롬프트를 하네스 내부에서 그대로 사용할 수 있다.

```text
너는 스킬 컴파일러다.
아래의 플랫폼 중립 스킬 원본 MD를 기반으로 대상 플랫폼용 최종 스킬 MD를 생성하라.

대상 플랫폼: {{target_platform}}
스킬 이름: {{skill_name}}
대상 경로: {{target_path}}
출력 형식: {{output_format}}
추가 제약: {{additional_constraints}}

핵심 규칙:
1. 원본 MD의 절대 원칙을 삭제하거나 약화하지 마라.
2. 증거 없는 PASS 금지 규칙을 최상단에 유지하라.
3. 테스트 상태값은 PASS, FAIL, BLOCKED, NOT TESTED, NOT APPLICABLE만 허용하라.
4. 관리자 페이지 검수 규칙을 반드시 포함하라.
5. 권한 테스트 규칙을 반드시 포함하라.
6. 화면 표시와 기능 작동을 분리하라.
7. 실행하지 않은 기능을 완료 또는 작동 확인으로 표현하지 마라.
8. 대상 플랫폼에 맞는 호출 방식과 안전 규칙을 추가하라.
9. 특정 기술 스택을 임의로 확정하지 마라.
10. 최종 결과는 대상 플랫폼에서 바로 사용할 수 있는 Markdown으로 출력하라.

원본 MD:
{{blueprint_md}}
```
