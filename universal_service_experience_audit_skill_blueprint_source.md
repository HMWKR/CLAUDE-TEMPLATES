# Universal Project Experience Audit Skill Blueprint Source

> 이 문서는 최종 `SKILL.md` 자체가 아니라, 내부 하네스가 Claude용 Skill, Codex용 Skill, 사내 에이전트용 Skill, QA 에이전트용 Prompt, PRD 검수 에이전트 등으로 변환할 수 있도록 만든 **스킬 생성용 원천 명세 MD**이다.
>
> 목적은 “어떠한 프로젝트든” 사용자 경험, 서비스 경험, 관리자 경험, 기능 테스트, 예외 처리, 운영 리스크를 초정밀하게 검수할 수 있는 스킬을 생성하는 것이다.

---

## 0. 이 MD의 사용 방식

이 문서는 런타임에서 그대로 사용자에게 보여주는 지침이 아니다.
내부 하네스는 이 문서를 읽고, 대상 에이전트와 실행 환경에 맞는 최종 스킬 지침을 생성해야 한다.

### 이 문서의 역할

- 범용 검수 스킬의 목적 정의
- 어떤 프로젝트에도 적용 가능한 검수 관점 정의
- UI가 있는 프로젝트와 UI가 없는 프로젝트 모두를 검수하기 위한 공통 프레임 정의
- Claude용, Codex용, 사내 에이전트용 등으로 변환 가능한 구조 제공
- 스킬 생성 시 반드시 포함해야 할 원칙, 절차, 출력 형식, 테스트 기준 제공

### 이 문서가 아닌 것

- 특정 서비스 전용 QA 문서가 아니다.
- 특정 UI/UX 체크리스트만 담은 문서가 아니다.
- Claude 또는 Codex 전용 최종 `SKILL.md`가 아니다.
- 특정 기술 스택 전용 개발 지침이 아니다.
- 보안 전문 침투 테스트 매뉴얼이 아니다.

## 2. 생성될 스킬의 핵심 목적

하네스는 이 블루프린트를 기반으로 다음 목적을 가진 최종 스킬을 생성해야 한다.

> 사용자가 제공한 프로젝트 정보, 기획서, 화면, 코드, 저장소, 문서, API 설명, 관리자 기능, 자동화 흐름, 챗봇 흐름 등을 바탕으로 사용자의 전체 경험과 운영자의 전체 경험을 초정밀하게 검수하고, 개선점과 테스트 케이스를 실행 가능한 수준으로 도출한다.

생성될 스킬은 다음을 반드시 수행해야 한다.

1. 프로젝트 유형을 먼저 분류한다.
2. UI가 있는 경우 모든 사용자 화면과 관리자 화면을 검수한다.
3. UI가 없는 경우에도 입력 경험, 출력 경험, 실패 경험, 운영 경험, 로그, 복구 흐름을 검수한다.
4. 사용자가 목적을 달성하는 전체 여정을 검수한다.
5. 관리자, 운영자, 내부 사용자, 시스템 관리자의 여정을 검수한다.
6. 기능 존재 여부가 아니라 실제 사용 가능성과 실패 복구 가능성을 검수한다.
7. 모든 핵심 기능에 대해 정상, 예외, 실패, 엣지케이스 테스트를 설계한다.
8. 정보가 부족한 부분은 임의로 채우지 않고 명확하게 “검증 불가”로 분리한다.
9. 발견 사항마다 우선순위, 근거, 개선 방향, 테스트 방법을 제공한다.
10. 최종적으로 개발자, 디자이너, PM, QA, 운영자가 바로 사용할 수 있는 결과물을 출력한다.

## 3. 범용 프로젝트 분류 규칙

생성될 스킬은 검수를 시작하기 전에 프로젝트를 다음 관점으로 분류해야 한다.

### 3.1 인터페이스 유형

프로젝트에는 화면이 없을 수도 있다. 따라서 “페이지”만 기준으로 검수하면 안 된다.
다음 인터페이스를 모두 고려한다.

- 웹 화면
- 모바일 화면
- 관리자 화면
- CLI 명령어
- API 엔드포인트
- 챗봇 대화 흐름
- 프롬프트 입력/출력
- 자동화 트리거
- 이메일
- 문자
- 앱 푸시
- 슬랙/디스코드/카카오톡 알림
- 문서 양식
- 업로드/다운로드 파일
- 대시보드
- 로그 화면
- 설정 화면
- 승인/반려 화면
- 결제/정산 화면
- 고객센터/문의 흐름
- 백오피스 운영 흐름

### 3.2 프로젝트 유형

하네스가 생성하는 스킬은 다음 중 하나 이상으로 프로젝트를 분류해야 한다.

- 사용자용 앱 또는 웹사이트
- 관리자 페이지가 포함된 서비스
- SaaS 또는 B2B 도구
- 커머스 또는 결제 기반 서비스
- 커뮤니티 또는 콘텐츠 플랫폼
- 내부 운영 도구
- 반복 업무 자동화
- AI 챗봇
- AI 에이전트
- 프롬프트 시스템
- API 서비스
- 데이터 처리 파이프라인
- 문서 생성/검토 시스템
- 상담/신청/예약/접수 시스템
- 오프라인 서비스와 연결된 디지털 프로세스
- 복합형 프로젝트

### 3.3 사용자 역할

생성될 스킬은 최소한 다음 역할을 탐색해야 한다.

- 비회원
- 회원
- 유료 회원
- 무료 사용자
- 관리자
- 슈퍼 관리자
- 운영자
- 승인자
- 검토자
- 판매자
- 구매자
- 제휴사
- 고객센터 담당자
- 내부 직원
- 외부 파트너
- 시스템 또는 자동화 봇
- API 클라이언트
- 개발자 사용자

역할이 명시되지 않은 경우, 스킬은 “가능한 역할 후보”와 “확인 질문”을 분리해서 작성해야 한다.

## 4. 절대 원칙

하네스가 생성하는 최종 스킬에는 아래 원칙이 반드시 포함되어야 한다.

### 4.1 자기합리화 금지

최종 스킬은 근거 없이 다음과 같은 표현을 하지 않도록 만들어야 한다.

- “충분합니다.”
- “문제 없어 보입니다.”
- “대체로 괜찮습니다.”
- “사용자가 이해할 것입니다.”
- “운영자가 알아서 처리할 수 있습니다.”
- “나중에 해도 됩니다.”
- “일반적인 수준입니다.”

위와 같은 판단이 필요한 경우에는 반드시 다음을 함께 제시하게 해야 한다.

- 판단 근거
- 검증 조건
- 반례 가능성
- 추가 확인 방법
- 테스트 기준

### 4.2 임의 판단 금지

정보가 없으면 추정하지 않게 해야 한다.

잘못된 방식:

- “로그인 기능은 당연히 있을 것이다.”
- “관리자는 회원 관리를 할 것이다.”
- “에러 처리는 기본적으로 되어 있을 것이다.”
- “사용자는 버튼 의미를 알 것이다.”

올바른 방식:

- “로그인 기능 존재 여부가 제공되지 않아 검증 불가”
- “관리자 권한 범위가 명시되지 않아 권한 테스트 필요”
- “에러 상태 화면이 제공되지 않아 누락 가능성 있음”
- “버튼 라벨만으로 사용자가 결과를 예측할 수 있는지 확인 필요”

### 4.3 발견 사항 분류 의무

모든 지적은 다음 중 하나로 분류해야 한다.

- 확인된 문제
- 강한 리스크
- 확인 필요
- 정보 부족으로 검증 불가
- 개선 제안
- 테스트 필요 항목

### 4.4 “검수 완료” 표현 제한

모든 범위를 실제로 다루기 전까지 “검수 완료”라고 표현하지 않아야 한다.
정보가 부족한 경우에는 반드시 다음과 같이 표현해야 한다.

> 제공된 정보 기준 1차 검수

또는

> 현재 자료 기준 부분 검수

### 4.5 사용자 목적 달성 중심

기능이 존재하는지 여부만 확인하지 않게 해야 한다.
항상 다음 질문을 기준으로 검수해야 한다.

- 사용자가 이 기능을 왜 써야 하는지 이해하는가?
- 사용자가 처음에 무엇을 해야 하는지 아는가?
- 사용자가 중간에 포기할 만한 지점은 없는가?
- 사용자가 실수했을 때 복구할 수 있는가?
- 완료 후 다음 행동이 명확한가?
- 실패했을 때 이유와 해결책을 알 수 있는가?
- 관리자 또는 운영자는 이 행동의 결과를 추적할 수 있는가?
- 로그, 상태, 알림, 권한, 복구 흐름이 연결되어 있는가?

## 5. 스킬 생성 시 포함해야 할 핵심 검수 프레임

하네스가 만드는 최종 스킬은 다음 8개 프레임을 모두 포함해야 한다.

### 5.1 Service Experience Design 검수

서비스 경험 설계는 화면 안의 UI뿐 아니라 사용자가 서비스를 알게 되고, 가입하고, 사용하고, 실패하고, 문의하고, 재방문하고, 해지하거나 종료하는 전체 흐름을 검수한다.

검수 질문:

- 사용자는 처음 진입 시 서비스의 핵심 가치를 이해하는가?
- 사용자는 자신의 다음 행동을 즉시 알 수 있는가?
- 가입, 인증, 설정, 결제, 사용, 완료, 문의 흐름이 자연스럽게 연결되는가?
- 화면 밖의 알림, 이메일, 고객센터, 운영 처리까지 경험이 이어지는가?
- 에러 또는 지연이 발생했을 때 사용자가 방치되지 않는가?
- 정책, 가격, 제한사항, 환불, 데이터 처리 조건이 명확한가?

### 5.2 User Journey Optimization 검수

각 사용자 역할별로 목표 달성 과정을 단계별로 나누어 검수한다.

각 여정에는 다음 항목이 있어야 한다.

- 여정명
- 사용자 역할
- 사용자 목표
- 시작 지점
- 종료 지점
- 단계별 행동
- 필요한 입력값
- 시스템 피드백
- 이탈 가능 지점
- 불필요한 단계
- 줄일 수 있는 입력
- 자동화 가능한 부분
- 실패 시 복구 흐름
- 관리자 또는 운영자 개입 지점
- 개선된 이상 흐름

### 5.3 Frictionless UX 검수

사용자가 귀찮아하거나, 불안해하거나, 헷갈리거나, 포기하게 되는 마찰을 찾는다.

마찰 유형:

- 불필요한 입력
- 반복 입력
- 모호한 버튼명
- 다음 단계 불명확
- 과도한 선택지
- 복잡한 인증
- 늦게 공개되는 가격 또는 조건
- 불친절한 에러 메시지
- 취소/수정/복구 불가
- 결제 실패 복구 불가
- 로딩 상태 부재
- 빈 상태 안내 부재
- 권한 없음 이유 부재
- 관리자 승인 상태 미표시
- 고객센터 경로 숨김
- 상태값 불명확
- 너무 긴 문장
- 어려운 용어

각 마찰은 다음 형식으로 출력해야 한다.

```text
- 위치:
- 마찰 유형:
- 현재 문제:
- 사용자가 느낄 감정:
- 왜 문제가 되는가:
- 개선 방향:
- 검증 방법:
- 우선순위:
```

### 5.4 Usability 검수

사용자가 쉽고 빠르게 목적을 달성할 수 있는지 검수한다.

검수 질문:

- 사용자가 3초 안에 핵심 행동을 알 수 있는가?
- 버튼명은 결과를 예측하게 만드는가?
- 필수 입력과 선택 입력이 구분되는가?
- 입력 오류가 구체적으로 설명되는가?
- 삭제, 결제, 승인, 반려 같은 위험 행동에 확인 단계가 있는가?
- 현재 위치와 진행 단계가 명확한가?
- 화면 간 용어가 일관적인가?
- 모바일, 태블릿, 데스크톱에서 사용할 수 있는가?
- 키보드만으로 조작 가능한가?
- 중요한 정보와 덜 중요한 정보가 시각적으로 구분되는가?

### 5.5 CX 검수

고객 경험은 화면 밖의 경험까지 포함한다.

검수 질문:

- 가입 전 기대와 실제 사용 경험이 일치하는가?
- 가격, 정책, 조건, 환불, 제한사항이 명확한가?
- 사용자가 불안해할 순간에 충분한 설명이 있는가?
- 문의 경로가 적절한 위치에 있는가?
- 문제 발생 시 사용자가 어디로 가야 하는지 알 수 있는가?
- 알림 문구가 친절하고 정확한가?
- 관리자 처리 지연 시 사용자에게 상태가 안내되는가?
- 서비스가 사용자를 방치하는 구간은 없는가?
- 신뢰를 해치는 요소가 있는가?
- 재방문 이유가 명확한가?

### 5.6 Admin / Operator Experience 검수

관리자 페이지는 부가 기능이 아니라 별도의 핵심 제품으로 다뤄야 한다.

검수 항목:

- 관리자 로그인
- 권한 구분
- 슈퍼 관리자와 일반 관리자 구분
- 회원 관리
- 콘텐츠 관리
- 주문/결제/환불 관리
- 문의 관리
- 신고 관리
- 승인/반려 처리
- 상태 변경
- 검색, 필터, 정렬
- 대량 처리
- 엑셀 다운로드 또는 데이터 내보내기
- 민감 정보 마스킹
- 변경 이력
- 감사 로그
- 작업 실패 복구
- 운영자 실수 방지
- 위험 작업 확인
- 권한 없는 접근 차단
- 사용자 화면 반영 여부
- 고객 문의 해결에 필요한 정보 제공 여부

관리자 기능은 다음 질문으로 검수해야 한다.

- 운영자가 가장 자주 하는 작업을 빠르게 할 수 있는가?
- 검색/필터/정렬이 운영 규모에 충분한가?
- 누가 언제 무엇을 변경했는지 알 수 있는가?
- 실수로 삭제, 승인, 환불, 차단할 위험은 없는가?
- 되돌리기 또는 복구 방법이 있는가?
- 민감 정보가 과도하게 노출되지 않는가?
- 고객 문의 해결에 필요한 맥락이 한 화면에 있는가?
- 관리자 작업 결과가 사용자 화면에 어떻게 반영되는지 명확한가?

### 5.7 State Coverage 검수

모든 화면, 기능, 흐름에는 상태가 있다.
스킬은 다음 상태가 누락되지 않았는지 검수해야 한다.

- 네트워크 실패 상태
- 저장 중 상태
- 저장 완료 상태
- 저장 실패 상태
- 삭제 확인 상태
- 삭제 완료 상태
- 삭제 실패 상태
- 결제 진행 상태
- 결제 완료 상태

- 사용자를 탓하지 않는가?

- 데이터 손실 위험이 없는가?
- 관리자에게 필요한 로그가 남는가?

### Step 1. 대상 에이전트 확인

프로젝트가 “앱”이라고만 적혀 있어도 화면만 검수하지 말고, 아래 표면을 함께 고려하는 스킬을 생성한다.

- 이메일
- 결제
- 문의
- 권한
- 로그
- 상태
- 실패 복구

- 외부 연동
- 테스트 케이스

프로젝트가 “자동화”라고만 적혀 있으면 다음 표면을 검수한다.

- 트리거

- 로그

- 실패/복구 흐름 포함
- UI가 없어도 경험 검수 수행

### Step 4. 출력 계약 삽입

최종 스킬에는 사용자가 결과를 바로 활용할 수 있도록 출력 형식을 고정한다.

필수 출력 섹션:

1. 제공된 정보 기준 검수 범위

3. 사용자 역할 목록

9. 상태별 누락 검수
10. 마찰 포인트 목록
11. 정보 부족으로 검증 불가한 항목
12. 누락 가능성이 있는 기능
13. 접근성/반응형/성능/권한 리스크
14. 출시 전 체크리스트

## 8. 하네스 입력 계약

하네스는 최종 스킬 생성 시 가능한 한 아래 입력을 받을 수 있어야 한다.
모든 값이 없어도 스킬이 작동해야 한다.

```yaml
input_contract:
  project_name: optional_string
  project_description: optional_string
  target_users: optional_list
  user_roles: optional_list
  project_type_hint: optional_string
  pages: optional_list
  admin_pages: optional_list
  features: optional_list
  user_flows: optional_list
  admin_flows: optional_list
  artifacts: optional_list
  repository_context: optional_object
  design_context: optional_object
  api_context: optional_object
  automation_context: optional_object
  chatbot_context: optional_object
  constraints: optional_list
  known_risks: optional_list
  audit_depth: optional_enum
  target_agent: required_enum
  output_language: optional_string
  evidence_mode: optional_enum
```

### 8.1 audit_depth 값

```yaml
audit_depth:
  quick:
    meaning: 핵심 리스크와 가장 큰 마찰만 검수
  standard:
    meaning: 주요 화면, 여정, 기능, 상태, 테스트를 검수
  ultra:
    meaning: 모든 가능한 표면, 상태, 예외, 관리자 경험, 테스트 케이스까지 초정밀 검수
```

기본값은 `ultra`로 한다.

### 8.2 evidence_mode 값

```yaml
evidence_mode:
  strict:
    meaning: 제공된 자료에서 확인 가능한 것과 추정 리스크를 엄격히 분리
  balanced:
    meaning: 자료 기반 검수와 합리적 리스크 탐색을 함께 수행
  exploratory:
    meaning: 아이디어 단계에서 누락 가능성과 확인 질문을 넓게 도출
```

기본값은 `strict`로 한다.

## 12. Claude용 스킬 생성 어댑터

이 장은 “Claude용 완성 스킬”이 아니라, 하네스가 Claude용 스킬을 만들 때 적용해야 할 변환 규칙이다.

### 12.1 Claude용 스킬 생성 목표

Claude용 최종 스킬은 일반 사용자가 프로젝트 설명, 기획서, 화면 캡처, 문서, 링크, 흐름 설명을 주었을 때 초정밀 검수 보고서를 생성하는 데 최적화한다.

### 12.2 Claude용 스킬 문서 구조 제안

하네스는 Claude용으로 변환할 때 다음 구조를 사용할 수 있다.

```markdown
# Universal Project Experience Audit

## When to use this skill
...

## Core role
...

## Non-negotiable rules
...

## Audit procedure
...

## Universal project classification
...

## Audit modules
...

## Output format
...

## Quality bar
...
```

### 12.3 Claude용 특화 지침

Claude용 최종 스킬에는 다음 성격을 강화한다.

- 사용자가 불완전한 아이디어만 제공해도 1차 검수를 시작한다.
- 정보 부족을 정중하지만 명확하게 분리한다.
- 화면이 없으면 사용자 여정, 입력/출력, 실패 경험, 관리자 경험 중심으로 검수한다.
- 긴 결과물은 섹션별로 정리한다.
- 질문은 마지막에 가장 중요한 5~10개만 둔다.
- 사용자에게 “자료를 더 주세요”만 말하고 멈추지 않는다.
- 제공 자료 기준으로 가능한 검수와 불가능한 검수를 분리한다.

### 12.4 Claude용 출력 강도

Claude용 최종 스킬은 다음 표현을 선호한다.

- “제공된 정보 기준으로 보면”
- “확인된 문제입니다”
- “강한 리스크입니다”
- “현재 자료만으로는 검증할 수 없습니다”
- “이 항목은 추가 확인이 필요합니다”
- “출시 전 테스트가 필요합니다”

다음 표현은 금지한다.

- “문제 없습니다”
- “괜찮습니다”
- “충분합니다”
- “일반적으로 그렇습니다”
- “아마 될 것입니다”

### 12.5 Claude용 생성 시 주의점

Claude는 설명형 답변에 강하므로, 최종 스킬은 보고서 품질과 reasoning structure를 강조한다.
그러나 내부 추론 과정을 그대로 노출하라는 지시를 넣지 않는다.
대신 “근거 요약”, “판단 기준”, “확인 방법”을 출력하게 한다.

## 13. Codex용 스킬 생성 어댑터

이 장은 “Codex용 완성 스킬”이 아니라, 하네스가 Codex용 스킬을 만들 때 적용해야 할 변환 규칙이다.

### 13.1 Codex용 스킬 생성 목표

Codex용 최종 스킬은 코드 저장소, 앱 구조, 라우트, 컴포넌트, 테스트 파일, 관리자 페이지 구현, API, 상태 관리, 에러 처리, 접근성, 테스트 커버리지를 실제 코드 관점에서 검수하는 데 최적화한다.

### 13.2 Codex용 스킬 문서 구조 제안

하네스는 Codex용으로 변환할 때 다음 구조를 사용할 수 있다.

```markdown
# Universal Project Experience Audit for Codebases

...

## Product experience audit protocol
```

### 13.3 Codex용 특화 지침

Codex용 최종 스킬에는 다음 성격을 강화한다.

- 먼저 저장소 구조를 파악한다.
- 라우트, 페이지, 컴포넌트, API, 상태 관리, 인증, 관리자 영역을 찾는다.
- 실제 구현된 화면과 기능을 기준으로 검수한다.
- 테스트 파일이 있으면 테스트 커버리지와 누락 케이스를 비교한다.
- 화면이 없어도 CLI, API, 워크플로우, 백엔드 상태, 로그, 에러 처리, 권한 처리를 검수한다.
- 코드 수정은 사용자가 명시적으로 요청한 경우에만 한다.
- 검수만 요청받았으면 보고서와 테스트 제안만 제공한다.
- 테스트 실행이 가능한 환경이면 기존 테스트 명령을 확인하고 결과를 근거로 삼는다.
- 테스트를 실행하지 못하면 실행하지 못한 이유와 대신 확인한 근거를 명시한다.

## 16. 우선순위 기준

최종 스킬에는 아래 기준이 반드시 포함되어야 한다.

### P0

즉시 수정해야 하는 치명적 문제.

예:

- 결제 오류
- 데이터 유실
- 권한 우회
- 개인정보 노출
- 핵심 기능 사용 불가
- 관리자 실수로 복구 불가능한 손상
- 잘못된 사용자에게 데이터 노출
- AI가 위험한 작업을 확인 없이 수행

### P1

출시 전 반드시 수정해야 하는 주요 문제.

예:

- 가입, 로그인, 결제, 신청, 작성 등 핵심 흐름에서 큰 불편
- 실패 후 복구 불가
- 관리자 처리 흐름 불명확
- 중요한 상태 누락
- 사용자가 높은 확률로 이탈할 문제
- 운영자가 고객 문의를 해결할 수 없는 정보 구조

### P2

사용성과 운영 품질을 크게 높이는 개선.

예:

- 문구 개선
- 단계 축소
- 버튼 위치 개선
- 안내 강화
- 검색/필터 개선
- 빈 상태 개선
- 관리자 대량 처리 개선
- 테스트 케이스 보강

### P3

있으면 좋은 개선.

예:

- 세부 인터랙션 개선
- 시각적 완성도 향상
- 부가 기능
- 고급 편의 기능
- 개인화

## 17. 품질 기준

하네스가 만든 최종 스킬은 아래 품질 기준을 만족해야 한다.

### 17.1 결과물 품질

- 추상적인 조언만 하지 않는다.
- 모든 문제에는 위치, 영향, 개선 방향, 테스트 방법이 있다.
- 모든 주요 문제에는 우선순위가 있다.
- 정보 부족 항목은 별도로 분리된다.
- 사용자 경험과 관리자 경험이 모두 포함된다.
- 기능 테스트와 상태 테스트가 모두 포함된다.
- 실제 작업자가 바로 사용할 수 있는 요구사항이 포함된다.

### 17.2 좋은 출력 예시

```text
회원가입 2단계에서 휴대폰 인증 실패 시 재시도 버튼과 오류 사유가 없으면 사용자가 이탈할 수 있다.
개선 방향: “인증번호가 일치하지 않습니다. 다시 입력하거나 인증번호를 재발송해 주세요.”라는 문구와 재발송 버튼을 제공한다.
테스트 방법: 잘못된 인증번호, 만료된 인증번호, 재발송 후 기존 번호 입력, 네트워크 실패를 각각 테스트한다.
우선순위: P1
```

```text
관리자 회원 상세 페이지에서 결제 상태, 문의 이력, 차단 여부가 분리되어 있으면 운영자가 문제를 해결하기 어렵다.
개선 방향: 회원 상세 페이지에 결제 내역, 문의 이력, 상태 변경 로그를 탭 구조로 통합한다.
테스트 방법: 특정 회원의 결제 실패 문의가 접수되었을 때 운영자가 한 화면에서 결제 상태, 최근 문의, 관리자 메모를 확인할 수 있는지 검증한다.
우선순위: P1
```

### 17.3 나쁜 출력 예시

```text
UX를 개선하세요.
```

## 18. 하네스 변환 시 금지 사항

하네스는 이 블루프린트를 최종 스킬로 변환할 때 다음을 하지 않아야 한다.

- 특정 프로젝트 유형만 검수 가능한 형태로 축소하지 않는다.
- UI/UX 화면 검수만 남기고 관리자/운영/테스트를 제거하지 않는다.
- “정보 부족하면 질문만 하라”는 형태로 만들지 않는다.
- 사용자가 자료를 덜 줬다는 이유로 검수를 멈추게 만들지 않는다.
- 테스트 케이스 생성을 선택 사항으로 만들지 않는다.
- 관리자 페이지 검수를 부가 항목으로 격하하지 않는다.
- 상태별 검수를 생략하지 않는다.
- 권한과 실패 복구를 생략하지 않는다.
- “완료” 표현을 쉽게 허용하지 않는다.
- AI가 임의로 가정하고 채워 넣도록 만들지 않는다.

## 20. 최종 스킬 생성용 압축 지시문

하네스가 짧은 지시문으로 최종 스킬을 생성해야 하는 경우, 아래 내용을 사용한다.

```text
이 블루프린트를 기반으로, 대상 에이전트가 어떤 프로젝트든 사용자 경험, 서비스 경험, 사용자 여정, 관리자 경험, 기능 테스트, 예외 상태, 실패 복구, 권한, 알림, 로그, 운영 리스크를 초정밀하게 검수할 수 있는 스킬을 생성하라.

최종 스킬은 UI가 있는 프로젝트뿐 아니라 자동화, 챗봇, 프롬프트, API, 데이터 워크플로우, 내부 운영 도구에도 적용 가능해야 한다.

최종 스킬은 절대 자기합리화하지 않고, 임의 판단하지 않으며, 확인된 사실과 추정 리스크와 검증 불가 항목을 분리해야 한다.

최종 스킬은 모든 발견 사항에 위치, 문제, 영향, 개선 방향, 테스트 방법, 우선순위를 포함하게 해야 한다.

최종 스킬은 모든 핵심 기능에 대해 정상, 예외, 실패, 엣지케이스, 권한, 상태, 알림, 관리자 반영, 로그, 복구 테스트를 생성하게 해야 한다.

최종 스킬은 정보가 부족할 때 검수를 멈추지 않고, 제공된 정보 기준 1차 검수를 수행한 뒤 부족한 정보를 별도 목록으로 정리하게 해야 한다.

최종 스킬은 모든 범위를 실제로 다루기 전까지 “검수 완료”라고 말하지 않게 해야 한다.
```

## 21. 하네스 구현자를 위한 체크리스트

이 블루프린트로 스킬을 생성한 뒤, 생성 결과가 아래 조건을 만족하는지 확인한다.

- [ ] 특정 프로젝트 유형에만 묶여 있지 않다.
- [ ] UI 없는 프로젝트도 검수할 수 있다.
- [ ] 관리자/운영자 경험을 포함한다.
- [ ] 사용자 여정과 서비스 경험 설계를 포함한다.
- [ ] Frictionless UX와 Usability를 포함한다.
- [ ] CX를 포함한다.
- [ ] 기능별 테스트 케이스 생성을 포함한다.
- [ ] 상태별 검수를 포함한다.
- [ ] 에러, 빈 상태, 로딩, 권한 없음, 실패 복구를 포함한다.
- [ ] 정보 부족 시 임의 판단하지 않는다.
- [ ] 추정 리스크와 확인된 문제를 분리한다.
- [ ] 우선순위 기준이 있다.
- [ ] 개발자/디자이너/QA/운영자 전달용 요구사항이 있다.
- [ ] Claude용 변환 시 설명형 검수에 적합하다.
- [ ] Codex용 변환 시 저장소/코드/테스트 검수에 적합하다.
- [ ] “검수 완료” 표현을 쉽게 허용하지 않는다.

## 22. 이 블루프린트의 핵심 문장

최종 스킬의 중심 문장은 다음과 같아야 한다.

> 이 스킬은 프로젝트의 형태와 상관없이 사용자가 목적을 달성하는 전체 경험, 운영자가 서비스를 관리하는 전체 경험, 기능이 정상/예외/실패 상황에서 작동하는지를 초정밀하게 검수한다. 정보가 부족하면 임의로 판단하지 않고, 제공된 정보 기준으로 가능한 검수와 검증 불가 항목을 분리한다.
