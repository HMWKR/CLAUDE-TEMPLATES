# 인사이트 분배 프로토콜 (범용)

> 프로젝트별 에이전트 매핑/도메인 날리지 경로는 해당 프로젝트의 `.claude/rules/`에 정의한다.
> **감시 에이전트**: `/insight-check` (insight-sentinel 스킬)로 미기록 인사이트 스캔 가능.
> **자동 감시**: Stop 훅에서 세션 종료 시 인사이트 기록 여부 자동 확인.
> **에이전트 매핑 워크플로우**: `~/.claude/rules/agent-mapping.md` 참조. 인사이트 기록은 매핑/Spawn 양쪽 워크플로우에서 동일하게 의무.

## 트리거 조건

작업 중 아래 상황이 발생하면 이 프로토콜을 실행한다:
- 새로운 기술적 발견 또는 의사결정이 있을 때
- 버그 원인/해결 방법이 다른 영역에도 영향을 줄 때
- 외부 API/라이브러리에 대한 검증된 지식이 생겼을 때
- 아키텍처 변경이 여러 계층에 영향을 미칠 때

## 실행 절차

### 1. 인사이트 식별
작업 완료 또는 중요 결정 시점에서 판단:
- 이 발견/결정이 **다른 영역에도 영향**을 미치는가?
- 이 지식이 **반복 사용**될 가능성이 있는가?
- 이것이 **기존 인사이트와 모순**되는가?

### 2. 관련 영역 매핑
프로젝트의 `.claude/rules/`에 에이전트 매핑 테이블이 있으면 참조한다.
없으면 작업 영역(FE/BE/AI/QA/기획 등)을 기준으로 교차 영향을 판단한다.

### 3. 사용자에게 확인
```
"[인사이트 요약] — 도메인 날리지에 기록할까요?"
옵션:
1. "전부 기록" — 관련된 모든 영역 파일에 추가
2. "선별 기록" — 직접 관련된 영역만
3. "기록 안 함" — 이번은 스킵
```

### 4. 기록 형식
```markdown
## {날짜}: {제목}
- **발견**: {무엇을 발견/결정했는지}
- **이유**: {왜 그렇게 했는지}
- **영향**: {다른 영역에 미치는 영향}
- **출처**: {어떤 작업 중 발견했는지}
- **다음**: {후속 작업}
```

기록 경로는 프로젝트별 `.claude/rules/`에서 정의한 도메인 날리지 디렉토리를 따른다.

### 5. 워크플로우별 기록 책임자

| 워크플로우 | 인사이트 기록자 | 시점 |
|---|---|---|
| **매핑** (메인 Claude가 페르소나로 변신) | 메인 Claude | 작업 완료 직후, 같은 페르소나로 그대로 기록 |
| **Spawn** (서브에이전트 위임) | 서브에이전트 (위임 시 명시) → 또는 메인 Claude (회수 후) | 서브에이전트 보고 회수 직후 메인이 정리 |

매핑 방식은 같은 페르소나가 발견·기록을 모두 수행하므로 일관성이 높다.
Spawn 방식은 서브에이전트의 보고를 메인이 받아 별도로 인사이트 파일에 정리하는 단계가 필요하다.

## 인사이트 관리 (수정/삭제/개선)

### 수정 트리거
- 기존 인사이트와 **모순**되는 새 발견이 있을 때
- 코드가 변경되어 기존 인사이트가 **더 이상 유효하지 않을 때**
- 더 나은 접근법이 발견되어 기존 인사이트를 **개선**할 수 있을 때

### 수정 절차
1. 해당 인사이트 파일을 읽어 모순/무효 항목 식별
2. 사용자에게 알림 후 승인 시:
   - **수정**: `[수정 {날짜}]` 태그 + 업데이트
   - **삭제**: 항목 제거 + 삭제 사유 기록
   - **개선**: 기존 내용 보강 + `[개선 {날짜}]` 태그

### 정기 검토
5회 이상의 인사이트가 쌓인 파일은 작업 시작 시:
- 중복 항목 병합 제안
- 오래된 항목(30일+) 유효성 확인 제안
- 모순 항목 해결 제안
