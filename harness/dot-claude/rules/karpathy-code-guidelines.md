# Karpathy Code Guidelines — 코드 작업 시 반드시 준수

> **코드 작성·수정·리팩토링·디버깅 시 아래 4원칙을 무조건 준수한다.** (사용자 명시 지침, 2026-06-09)
> 출처: `andrej-karpathy-skills:karpathy-guidelines` (원문: 해당 플러그인 SKILL.md). LLM 코딩 실수 감소용.
> Tradeoff: 신중함 > 속도. **사소한 작업은 판단껏** 적용 강도 조절.
> 적용 범위: 프로덕션 코드·테스트·스크립트 편집. 단순 질문/설명/문서는 제외 가능.

## 1. Think Before Coding — 가정을 드러내라
- 구현 전 **가정을 명시**한다. 불확실하면 질문한다.
- 해석이 여러 개면 silently 고르지 말고 **제시**한다.
- 더 단순한 길이 있으면 말하고 **push back** 한다.
- 불명확하면 멈추고 — 무엇이 혼란스러운지 명명하고 — 질문한다.

## 2. Simplicity First — 문제를 푸는 최소 코드, 투기적 코드 금지
- 요청 범위 밖 기능 금지. 단발 코드에 추상화 금지.
- 요청하지 않은 "유연성/설정성" 금지. **불가능한 시나리오**용 에러 처리 금지.
- 200줄을 50줄로 줄일 수 있으면 다시 쓴다.
- 자문: "시니어 엔지니어가 이거 과하다고 할까?" → yes면 단순화.

## 3. Surgical Changes — 건드려야 할 것만, 내 똥만 치운다
- 기존 코드/주석/포맷을 임의로 "개선"하지 않는다. **안 깨진 것 리팩토링 금지.**
- 기존 스타일 유지(내가 다르게 했을지라도). 무관한 dead code는 **언급만**, 삭제 금지.
- 내 변경이 만든 orphan(import/변수/함수)만 제거. 기존 dead code는 요청 시에만.
- **테스트: 바뀐 모든 줄이 사용자 요청에 직접 추적되어야 한다.**

## 4. Goal-Driven Execution — 성공 기준 정의 후 검증까지 루프
- 작업을 **검증 가능한 목표**로 변환:
  - "검증 추가" → "잘못된 입력 테스트 작성 → 통과시킨다"
  - "버그 수정" → "재현 테스트 작성 → 통과시킨다"
  - "리팩토링 X" → "전후 테스트 통과 보장"
- 멀티스텝은 간단한 plan 명시: `1.[단계]→검증:[체크]  2.… 3.…`
- 강한 성공 기준 = 독립 루프 가능. 약한 기준("작동되게")은 계속 확인을 부른다.
