## 표준 오케스트레이션 파이프라인 — 지휘·교차검증·main 승인 (사용자 설계 2026-07-06, 교정 반영)
멀티에이전트 실작업의 표준 흐름. **main도 이 형식을 따른다**(단 사소·단일스레드·단순질문·가역 작업은 면제 — 위 적극활용 가드레일).
1. **계획(main 지휘)**: 작업 그래프·분해를 main이 설계. DoD·검증기준은 `verification-templates/`에서 선택해 `verify-lock`으로 고정 — **main 자유저작 금지**.
2. **계획 검수**: 템플릿 커버리지 밖이면 codex(이종) 또는 상위 판정단이 그래프·DoD를 승인(원칙 4 — 그래프 내용 자동 신뢰 금지).
3. **실행(teammates 병렬)**: 파일=기억 규약, result 원문 보존, write_scope 경계, 워커 외부 직접쓰기 금지.
4. **1차 검수(codex 교차벤더)**: 각 산출물을 실행 모델 밖에서 반박.
5. **종합(main)**: result 종합·충돌 해소(`[DECISION]`).
6. **최종 검증(독립 증거)**: 통합 결과를 DoD 대비 codex/독립 검증 + verify-lock tamper + 카나리. **main 자기채점 아님 — 증거는 main 밖에서.**
7. **최종 승인(main)**: 6의 외부 증거에 근거해 go/no-go. **승인권 = main, 검증 증거 = 밖**("판단은 main, 채점은 밖").
- **실패 경로**: 동일 실패 K=3회 → 접근 강제 변경 → 상위 모델 에스컬레이션(발산 차단기). 진전 없는 반복엔 무제한 토큰 미적용.
- **고위험**: 검수 1명 대신 관점 분산(정확성·보안·재현) 또는 판정단 N(일부 이종)+불확실성 보존(만장일치 아니면 등급 보존).
- **실행 안정성(2026-07-07 스모크 교훈)**: 위 교차벤더 검수의 *지점*(2·4·6)은 옳으나, **codex를 워크플로 에이전트에 중첩 호출하면 stall-prone**(codex 세션 분 단위 > 무진전 타임아웃)이다. 그래서 `conductor-verify.js`는 내부를 **Claude 적대검수**로 돌리고, **결정적 교차벤더 codex 게이트는 워크플로 반환 후 main이 직접 호출**한다(직접 호출은 안정 실증). 원칙 불변: 검증증거는 실행 모델 밖(codex), 판단은 main.
