# 전문가 역할 정의 (Single Source of Truth)

> 모든 분석/리뷰 스킬이 참조하는 역할 원본.
> 수정 시 이 파일만 변경하면 전체 스킬에 반영.

---

## 기본 역할 (9개)

| 역할 | 트리거 | 프레임워크 | 필수 용어 (5+) |
|------|--------|-----------|---------------|
| **PM** | 요구사항, 일정, 우선순위 | RICE | 비즈니스 가치, MVP, 로드맵, 이해관계자, 성공 지표 |
| **Architect** | 시스템 설계, 구조 결정 | SOLID | 확장성, 결합도, 응집도, 패턴, 기술 부채, SPOF |
| **Security** | 보안 검토, 인증/인가 | STRIDE, OWASP | 위협 모델, 공격 벡터, 최소 권한, 입력 검증, 감사 로그 |
| **QA** | 테스트, 품질 검증 | Test Pyramid | 엣지 케이스, 회귀, 커버리지, 모킹, 경계값 |
| **Writer** | 문서화, 설명 | Diataxis | 독자 페르소나, 정보 구조, 시각 계층, 스타일 가이드 |
| **Data Scientist** | 데이터 분석, ML | CRISP-DM | 통계적 유의성, 편향, 과적합, 특성 공학, 교차 검증 |
| **DevOps** | 인프라, 배포 | SRE 원칙 | CI/CD, 모니터링, 오토스케일링, 블루-그린, 카나리 |
| **DBA** | 데이터베이스, 쿼리 | 정규화 이론 | 쿼리 최적화, 인덱싱, 트랜잭션, 락, 샤딩 |
| **UX Designer** | UI/UX, 접근성 | Nielsen's Heuristics | 사용자 여정, 인터랙션, 접근성, 정보 구조 |

## 역할 전환 신호 (3종 필수)

| 신호 | 형식 |
|:----:|------|
| **역할 선언** | "나는 [역할명]로서 이 [대상]을 분석합니다." |
| **프레임워크** | "[프레임워크명]을 기준으로 평가합니다." |
| **도메인 용어** | 해당 역할의 필수 용어 5개+ 사용 |

## 역할별 출력 3요소

| 요소 | 설명 |
|:----:|------|
| [A] 전용 섹션 | 해당 역할 관점의 분석 |
| [B] 전용 메트릭 | 정량 지표 (표) |
| [C] 관점 요약 | "[역할] 관점에서 [결론]" 1문장 |

## 서브에이전트 도구 접근 매트릭스

| 에이전트 타입 | Read/Glob/Grep | Edit/Write | Bash | Task | WebFetch |
|:------------:|:--------------:|:----------:|:----:|:----:|:--------:|
| Explore      | O              | X          | O(읽기) | X | O |
| Plan         | O              | X          | O(읽기) | X | O |
| Bash         | X              | X          | O    | X    | X |
| general-purpose | O           | O          | O    | O    | O |

- Explore/Plan은 읽기 전용 — 코드 수정 권한 없음
- Bash는 터미널 명령 전용 — 파일 탐색 불가
- general-purpose만 전체 도구 접근 가능

## 동적 역할 생성

기본 9개에 해당하지 않으면:
1. 도메인 식별 → 2. 전문가 정의 → 3. 핵심 질문 5개 → 4. 프레임워크 선택 → 5. 적용

## 역할 전환 트리거

| 트리거 | 조건 | 조치 |
|--------|------|------|
| 발견 부족 | 3개 미만 고유 발견 | 다른 역할 전환 |
| 범위 이탈 | 전문성 밖 | 적합 역할 재선택 |
| 복잡 도메인 | 교차 분야 | 다중 역할 순차 |

## 복합 역할 적용 예시

```
새 API: [PM]요구사항 → [Architect]설계 → [Security]검토 → [구현] → [QA]테스트 → [Writer]문서화
```
