# 확인 루프 프로토콜 (Confirmation Loop)

> 모든 스킬이 참조하는 사용자 확인 루프 공통 프로토콜.
> 스킬별 훅 오버라이드로 커스터마이즈하여 사용한다.

---

## 1. 핵심 원칙

- **Iron Law**: 사용자의 Yes 없이는 다음 단계로 넘어가지 않는다.
- **피드백 반영 필수**: 같은 내용을 단순 반복하지 않는다 -- 사용자 피드백 반영 후 재제시.
- **구체적 선택지**: 열린 질문 대신 AskUserQuestion으로 구체적 선택지를 제시한다.

## 2. 루프 구조 (Generic)

```
1. 결과를 {PRESENT_FORMAT}으로 제시

2. AskUserQuestion으로 확인 요청:
   - {CONFIRM_OPTIONS[0]} → 다음 {STEP_LABEL}로 진행
   - {CONFIRM_OPTIONS[1]} → 수정 루프 진입

3-A. 확인됨 → 다음 단계

3-B. 수정 필요 → 수정 서브루프:
   ├─ {MODIFY_OPTIONS_BUILDER}로 수정 선택지 제시 (AskUserQuestion)
   │   ├─ 선택지 선택 → 반영 후 1번 복귀
   │   ├─ 자유 텍스트 → 반영 후 1번 복귀
   │   └─ "다시 제시" → 핸들러 참조 (Section 4)
   └─ (Yes까지 루프 반복)
```

## 3. 수정 서브루프

1. 현재 출력 기준으로 수정 가능한 항목을 구체적 선택지로 제시한다.
2. 사용자가 선택지를 고르면 해당 항목만 수정 후 전체 결과를 재제시한다.
3. 사용자가 자유 텍스트로 피드백하면 내용을 반영 후 전체 결과를 재제시한다.
4. 사용자가 "다시 제시"를 요청하면 Section 4 핸들러로 분기한다.
5. 재제시된 결과에 대해 다시 Section 2의 확인 요청부터 반복한다.

## 4. "다시 제시" 키워드 핸들러

1. 사용자의 이전 피드백/컨텍스트를 종합하여 선택지를 재구성한다.
2. 재구성된 선택지를 AskUserQuestion으로 제시한다.
3. 사용자 선택을 대기한다.
4. 선택 후 반영하여 {PRESENT_FORMAT}으로 결과를 재제시한다.
5. Section 2의 확인 루프로 복귀한다.

## 5. 커스터마이제이션 훅

| 훅 | 기본값 | 설명 |
|:--:|--------|------|
| `PRESENT_FORMAT` | 자연어 요약 + 표 | 결과 제시 형식. 스킬별로 Progressive Table, 점수표 등으로 오버라이드 |
| `CONFIRM_OPTIONS` | ["맞습니다", "수정이 필요합니다"] | AskUserQuestion 확인 선택지 |
| `MODIFY_OPTIONS_BUILDER` | 수정 가능 항목을 자동으로 선택지화 | 수정 선택지를 생성하는 방식 |
| `STEP_LABEL` | "단계 N" | 단계 표기. "STEP N", "Phase N", "Turn N" 등으로 오버라이드 |

**스킬별 오버라이드 패턴:**

```markdown
## Confirmation Loop

> **프로토콜 참조**: `_core/confirmation-loop.md`

**훅 오버라이드:**
| 훅 | 값 |
|:--:|-----|
| `PRESENT_FORMAT` | [스킬별 형식] |
| `CONFIRM_OPTIONS` | [스킬별 선택지] |
| `MODIFY_OPTIONS_BUILDER` | [스킬별 빌더] |
| `STEP_LABEL` | [스킬별 표기] |
```

## 6. 안전 규칙

- **단순 반복 금지**: 피드백 미반영 재제시는 금지한다. 반드시 변경점을 명시한다.
- **반복 임계**: 수정 루프에서 3회 이상 반복 시 사용자에게 근본적 방향 재설정을 제안한다.
- **즉시 탈출**: 사용자가 "스킵", "그냥 진행", "중단" 등을 요청하면 즉시 루프를 탈출한다.
- **자가검증**: 각 루프 반복에서 사용자의 피드백을 정확히 반영했는지 자가검증한다.
