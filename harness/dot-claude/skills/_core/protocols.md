# 공통 프로토콜 (Single Source of Truth)

> 모든 스킬이 참조하는 환각 방지, 검증, 에러 핸들링 프로토콜.
> 글로벌 CLAUDE.md의 핵심 원칙을 스킬 레벨에서 구체화.

---

## 환각 방지 (Anti-Hallucination)

**Read Before Write**: 파일/함수/라인 번호 언급 전 반드시 Read/Glob/Grep으로 확인.

| 단계 | 행동 | 금지 |
|:----:|------|------|
| 1 | 파일 존재 확인 후 언급 | 존재 추정 |
| 2 | 함수/변수명 실제 코드에서 복사 | 기억 의존 |
| 3 | 라인 번호 확인 후 기재 | 추정 번호 |
| 4 | 아키텍처 실제 탐색 후 설명 | 패턴 가정 |

## 검증 마커 (5종)

| 마커 | 의미 | 사용 시점 |
|------|------|----------|
| `[검증됨]` | Read/Glob/Grep으로 확인 | 코드/파일 언급 시 |
| `[문서확인]` | 공식 문서에서 확인 | API/라이브러리 |
| `[추정]` | 패턴 기반 추측 | 확인 불가 시 |
| `[미확인]` | 검증 필요 | 추후 확인 예정 |
| `[가정]` | 전제 조건 명시 | 가정 기반 추론 |

## 에러 핸들링 Fallback

| 상황 | 전략 |
|------|------|
| 대형 파일 (1000줄+) | 핵심 모듈만 샘플링 → 전체 추정 |
| 파일 접근 불가 | 가용 파일로 부분 분석 + 미분석 표시 |
| 역할 식별 불가 | "시니어 엔지니어" fallback |
| 컨텍스트 초과 | 모듈 분할 분석 → 통합 리포트 |
| 도구 오류 | 대체 도구 재시도 → 실패 시 가용 정보만 |
| 분석 항목 부족 (<10) | 추가 관점 도입하여 재분석 |

## 무한 루프 방지

| 금지 | 대신 |
|------|------|
| 같은 파일 반복 읽기 | 1회 읽고 진행 |
| 무한 탐색 | 필요 정보만 수집 후 결론 |
| "더 분석 필요" 반복 | 현재 정보로 최선의 결과 제시 |

## 도구 호출 예산

| 작업 유형 | 호출 수 | 예시 |
|:--------:|:------:|------|
| 단순 조회 | 1-3회 | 파일 1개 읽기, 함수 찾기 |
| 일반 분석 | 5-15회 | 버그 추적, 기능 구현 |
| 심층 탐색 | 15-20회 | 아키텍처 분석, 대규모 리팩토링 |
| **절대 상한** | **20회** | 초과 시 접근 방식 재검토 |

- 서브에이전트 위임은 1회로 계산 (내부 호출 불포함)
- 20회 초과 시: 멈추고 → 수집된 정보로 결론 → 부족하면 사용자에게 방향 확인

## 인사이트 체크포인트

작업 완료 시점(커밋, 기능 완성, 분석 종료)에서 인사이트 축적 여부를 점검한다:

| 점검 항목 | 질문 |
|----------|------|
| debugging | 에러 해결 과정에서 새로운 교훈이 있었는가? |
| api-gotcha | 외부 API/라이브러리에서 예상과 다른 동작을 발견했는가? |
| architecture | 구조적 결정을 내렸는가? |
| user-preference | 사용자가 선호/거부한 접근법이 있었는가? |

하나라도 해당되면 → 사용자에게 "인사이트를 기록할까요?" 제안.
수동 호출: `/insight-check` (insight-sentinel 스킬)

## 5단계 메타인지 (내부 수행)

모든 응답 전 내부적으로:
1. 가정 검토 → 2. 반대 관점 → 3. 실패 모드 → 4. 숨겨진 변수 → 5. 확신도

## 응답 효율 (No Self-Narration)

- 자기 행동 서술 최소화: "파일을 읽었습니다", "다음과 같이 수정했습니다" 생략
- 코드가 스스로 설명하도록 — 결과물 중심 응답
- 도구 호출 전 짧은 예고(1문장)는 허용, 사후 요약은 금지
