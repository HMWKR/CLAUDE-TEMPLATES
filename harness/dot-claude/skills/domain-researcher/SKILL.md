# Domain Researcher — 도메인 날리지 자동 수집 파이프라인

## 핵심 철학

**추측하지 말고 검색하라. 검색되지 않으면 [추정]이라고 말하라.**

모든 사실 기반 정보에 신뢰도 태그를 붙인다:
- `[검색확인]` — web_search/web_fetch로 직접 확인한 정보 + 출처 URL
- `[내부지식]` — 학습 데이터 기반, 검증 안 됨
- `[추정]` — 확인되지 않은 추론, 근거를 함께 명시
- `[상충]` — 출처 간 정보가 다를 때 양쪽 모두 기록

---

## 실행 전 사전 작업

### 도메인 키워드 추출

사용자 메시지에서 추출:
- **도메인**: 펫, 교육, 의료, 부동산, 물류, F&B, 패션, 금융 등
- **서비스 유형**: 쇼핑몰, 플랫폼, SaaS, 앱, 마켓플레이스 등
- **특수 기술**: AI, 블록체인, IoT, AR/VR 등 (있는 경우)
- **지역**: 한국 기본, 글로벌/특정 국가 명시 시 반영

이 키워드들이 5개 모듈의 검색 쿼리에 자동 삽입된다.

### 실행 모드

- **전체 모드** (기본): 5개 모듈 전부 실행
- **선택 모드**: 사용자가 특정 모듈만 요청 ("시장 규모만 알아봐줘")
- **딥 모드**: 특정 모듈을 검색 횟수 2배로 심층 조사

---

## 5개 리서치 모듈

각 모듈의 상세 검색 전략과 출력 템플릿은 `references/research-modules.md`를 읽고 따른다.

### MODULE 1: 시장 분석 (market-analysis.md)

**목적**: 타겟 시장의 규모, 성장률, 핵심 트렌드를 정량적으로 파악한다.

**검색 전략** (3~5회):
```
검색 1: "{도메인} 시장 규모 {연도}" (예: "펫 의류 시장 규모 2025")
검색 2: "{도메인} 시장 성장률 CAGR"
검색 3: "{도메인} 산업 트렌드 {연도}"
검색 4: "{도메인} 온라인 시장 비중" (해당 시)
검색 5: "{도메인} TAM SAM SOM" (해당 시)
```

**출력 구조**:
- 시장 규모 (국내/글로벌, 연도 명시, 출처 URL)
- 성장률 (CAGR, 기간 명시)
- 핵심 트렌드 3~5개
- TAM/SAM/SOM 추정 (근거 포함)
- 계절성/주기성 (해당 시)
- 핵심 수치마다 [검색확인] 또는 [추정] 태그

**교차검증 규칙**:
- 시장 규모는 최소 2개 출처에서 확인
- 출처 간 20% 이상 차이 시 [상충] 태그 + 양쪽 수치 모두 기록
- 정부/공공기관 > 업계 협회 > 시장조사 기관 > 언론 기사 > 블로그 순으로 신뢰

### MODULE 2: 경쟁사 분석 (competitor-analysis.md)

**목적**: 기존 서비스를 조사하여 기능 비교 매트릭스를 만들고, 미해결 문제를 도출한다.

**검색 전략** (5~8회):
```
검색 1: "{도메인} {서비스유형} 서비스 추천" (예: "온라인 교육 플랫폼 추천")
검색 2: "{도메인} {서비스유형} 순위 비교"
검색 3~7: 개별 경쟁사명 + "리뷰" 또는 "기능" (상위 5개)
검색 8: "{도메인} 불만 불편 개선" (미해결 문제 탐색)
```

**출력 구조**:
- 경쟁사 목록 (5~10개, 국내 위주 + 해외 주요)
- 각 경쟁사: 서비스명, URL, 핵심 기능, 과금 모델, 강점/약점
- 기능 비교 매트릭스 (마크다운 테이블)
- 미해결 문제 3~5개 (= 우리의 차별화 기회)
- 사용자 불만 패턴 (리뷰에서 추출)

### MODULE 3: 기술 스택 리서치 (tech-stack-research.md)

**목적**: 해당 도메인에서 사용되는 핵심 기술, 오픈소스, API, 논문을 조사한다.

**검색 전략** (3~5회):
```
검색 1: "{도메인} 기술 스택 개발" (예: "이커머스 기술 스택")
검색 2: "{핵심기능} 오픈소스 라이브러리" (예: "실시간 채팅 오픈소스")
검색 3: "{핵심기능} API 서비스" (예: "결제 API 한국")
검색 4: "{특수기술} 논문 벤치마크" (AI 등 해당 시)
검색 5: "{도메인} SaaS 인프라 아키텍처" (해당 시)
```

**출력 구조**:
- 핵심 기술 목록 (각각: 이름, 용도, 장단점, 라이선스)
- 오픈소스 추천 (GitHub stars, 최근 업데이트 여부)
- 외부 API/서비스 (결제, 인증, 알림, 지도 등)
- 관련 논문/벤치마크 (AI/ML 관련 시)
- 기술 선택 추천 (근거 포함)

### MODULE 4: 도메인 데이터 (domain-data.md)

**목적**: 프로젝트의 핵심 엔티티에 대한 실제 데이터 구조를 수집한다. PetFit의 "견종별 체형 데이터"에 해당하는 것.

**검색 전략** (2~3회):
```
검색 1: "{핵심엔티티} 데이터 기준 표준" (예: "강아지 견종 체형 표준")
검색 2: "{핵심엔티티} 분류 체계 카테고리"
검색 3: "{핵심엔티티} 통계 데이터셋"
```

**도메인별 핵심 데이터 예시**:
| 도메인 | 핵심 엔티티 | 수집할 데이터 |
|--------|-----------|-------------|
| 펫 의류 | 견종 | 체중, 목둘레, 가슴둘레, 등길이 (견종별) |
| 교육 | 과목/커리큘럼 | 학습 시간, 난이도, 선수과목 관계 |
| 의료 예약 | 진료과/시술 | 평균 진료시간, 보험 적용 여부, 준비사항 |
| 부동산 | 매물 유형 | 평형 구간, 지역별 시세, 거래 유형 |
| F&B 예약 | 음식 카테고리 | 평균 식사 시간, 가격대, 예약률 |
| 피트니스 | 운동 종류 | 칼로리 소모, 장비, 난이도, 근육 부위 |

**출력 구조**:
- 핵심 엔티티 정의 + 분류 체계
- 실제 데이터 테이블 (MVP에 필요한 상위 10~20개)
- JSON 또는 SQL 시드 데이터 형식으로도 제공
- 데이터 출처 + 신뢰도 표시
- 개체별 편차/주의사항

### MODULE 5: 규제/법률 (regulation-compliance.md)

**목적**: 해당 도메인의 법적 요구사항과 컴플라이언스를 조사한다.

**검색 전략** (2~3회):
```
검색 1: "{도메인} 관련 법률 규제 한국" (예: "온라인 교육 관련 법률")
검색 2: "{서비스유형} 사업자 등록 신고 요건"
검색 3: "{도메인} 개인정보 데이터 규제"
```

**출력 구조**:
- 필수 법적 요건 (사업자 등록, 신고, 인허가)
- 개인정보보호법 적용 범위
- 도메인 특화 규제 (의료: 의료법, 금융: 전자금융거래법, 교육: 학원법 등)
- AI 관련 규제 (해당 시: EU AI Act, 국내 AI 기본법 동향)
- 소비자보호 의무 (청약 철회, 환불 정책)
- 해외 진출 시 추가 규제 (해당 시)

## 출처 신뢰도 위계

검색 결과의 출처를 다음 순서로 신뢰한다:

1. **Tier 1** (높음): 정부 통계(통계청, 산업부), 법률(법제처), 공공 데이터
2. **Tier 2** (중상): 업계 협회 보고서, 상장사 IR 자료, 학술 논문
3. **Tier 3** (중간): 시장조사 기관(Statista, 닐슨 등), 주요 언론(경제지)
4. **Tier 4** (중하): 테크 미디어, 업계 블로그, 프레스 릴리즈
5. **Tier 5** (낮음): 개인 블로그, 커뮤니티 글, SNS

Tier 1~2의 정보는 [검색확인]으로 표기.
Tier 3~4만 있으면 [검색확인]이되 "시장조사 기관 기준" 등 출처 유형 명시.
Tier 5만 있으면 [추정]으로 표기하고 "공식 통계 확인 필요" 주석.

## 교차검증 프로토콜

### 수치 교차검증
1. 핵심 수치(시장 규모, 성장률)는 최소 2개 출처에서 확인
2. 출처 간 차이가 20% 이내: 평균값 사용, 양쪽 출처 모두 기록
3. 출처 간 차이가 20% 초과: [상충] 태그, 양쪽 수치와 출처 모두 기록, 판단 보류
4. 1개 출처만 발견: [단일출처] 태그 + "교차검증 필요" 주석

### 정보 부재 처리
- 검색 결과가 없으면: "해당 정보를 공개 소스에서 확인하지 못했습니다" — "없다"고 단정하지 않음
- 오래된 데이터만 있으면: 연도 명시 + "최신 데이터 확인 필요" 주석
- 한국 데이터 없고 글로벌만 있으면: 글로벌 수치 기록 + 한국 비중 추정

## 스킬 체인 연동

### domain-researcher → project-bootstrapper 연동
domain-researcher의 산출물이 bootstrapper의 다음 단계에 직접 입력된다:

| domain-researcher 모듈 | bootstrapper 단계 | 연동 방식 |
|----------------------|-----------------|----------|
| market-analysis.md | STAGE 7 (사업계획서) | 시장 규모/성장률 → 재무 계획 근거 |
| competitor-analysis.md | STAGE 1 (에이전트) + STAGE 4 (와이어프레임) | 미해결 문제 → 차별화 기능 → 화면 설계 |
| tech-stack-research.md | STAGE 2 (CLAUDE.md) + STAGE 3 (아키텍처) | 기술 추천 → 기술 스택 확정 |
| domain-data.md | STAGE 5 (DB) + STAGE 6 (코드) | 데이터 구조 → DB 스키마 + 타입 정의 + 시드 데이터 |
| regulation-compliance.md | STAGE 2 (CLAUDE.md 금지사항) + STAGE 7 (사업계획서 리스크) | 법적 요건 → 금지사항 + 리스크 분석 |

### project-kickstart에서 자동 호출
kickstart 실행 시 도메인 날리지가 없으면 domain-researcher를 선행 호출할 수 있다.
순서: kickstart → domain-researcher(5모듈) → bootstrapper(7단계) → ZIP

## 주의사항

- 웹 검색 결과를 그대로 복사하지 않는다. 핵심 수치와 팩트만 추출하여 자체 문장으로 재구성
- 유료 보고서의 전문을 재현하지 않는다. 공개된 요약/프리뷰만 활용
- 경쟁사 분석에서 특정 서비스를 과도하게 비하하지 않는다. 객관적 기능 비교에 집중
- 법률 정보는 "참고용이며 법적 자문을 대체하지 않는다" 면책 조항을 반드시 포함
- 검색이 충분히 안 되는 영역은 솔직하게 "[정보 부족] 추가 조사 필요"로 표기
