# Deep Analysis Mode 전문가 프로필

> 이 파일은 deep-analysis-mode 스킬의 전문가 역할 프로필과 복합 시나리오를 포함합니다.
> 역할 전환(Role-Playing) 시 이 파일을 참조하여 전문가 관점을 적용합니다.

---

## 디버깅 전문가

### 전문 영역
- 루트 원인 분석 (Root Cause Analysis)
- 재현 절차 수립
- 이분 탐색 기법 (Git bisect 등)
- 메모리 누수 진단
- 레이스 컨디션 탐지

### 접근 방식
1. 증상 정확히 기술
2. 재현 최소 조건 식별
3. 가설 수립 → 실험 설계
4. 이분 탐색으로 원인 범위 축소
5. 수정 후 회귀 확인

### 방법론
1. **증상 수집**: 에러 메시지, 스택 트레이스, 재현 조건 수집
2. **가설 수립**: 가능한 원인 3개 이상 나열, 확률 순으로 정렬
3. **격리 테스트**: 이분 탐색으로 문제 범위를 좁히기
4. **근본 원인 확인**: 증상이 아닌 원인을 수정
5. **회귀 방지**: 동일 버그 재발 방지 테스트 작성

## 성능 엔지니어

### 전문 영역
- 프로파일링 (CPU, 메모리, I/O)
- 병목 식별 및 최적화
- 캐싱 전략 설계
- 비동기/병렬 처리 최적화
- 데이터베이스 쿼리 최적화

### 측정 지표
- 응답 시간 (P50, P95, P99)
- 처리량 (TPS, QPS)
- 리소스 사용률 (CPU, RAM, Disk I/O)
- 에러율

### 방법론
1. **프로파일링**: 병목 지점 식별 (CPU, 메모리, I/O, 네트워크)
2. **벤치마크 설정**: 개선 전 기준선 측정
3. **최적화 적용**: 가장 큰 영향을 미치는 병목부터 해결
4. **비교 분석**: 최적화 전후 정량적 비교
5. **트레이드오프 문서화**: 성능 vs 가독성, 메모리 vs 속도 등

## 시스템 아키텍트

### 전문 영역
- 마이크로서비스 vs 모놀리스 결정
- 이벤트 드리븐 아키텍처
- CQRS/이벤트 소싱
- API 설계 (REST, GraphQL, gRPC)
- 확장성 패턴 (수평/수직 확장)

### 설계 원칙
- 단일 책임 원칙 (SRP)
- 의존성 역전 (DIP)
- 인터페이스 분리 (ISP)
- 최소 권한 원칙
- 장애 격리 (Bulkhead)

### 방법론
1. **요구사항 분석**: 기능적 비기능적 요구사항 분리
2. **제약 조건 식별**: 기술적, 비즈니스, 인프라 제약
3. **패턴 선택**: 적합한 아키텍처 패턴 비교
4. **확장성 설계**: 수평 수직 확장 전략
5. **위험 완화**: 단일 실패 지점(SPOF) 제거, 폴백 전략

## 복합 분석 시나리오

### 시나리오 1: 프로덕션 성능 저하

```
Stage 1: 컨텍스트 수집
  → 모니터링 대시보드, 에러 로그, 최근 배포 이력

Stage 2: 전문가 선택
  → 성능 엔지니어 + 디버깅 전문가 (복합)

Stage 3: 가설
  A: 최근 배포의 N+1 쿼리 도입
  B: 트래픽 증가로 인한 리소스 포화
  C: 외부 API 응답 지연 전파

Stage 4: 증거 수집
  → slow query log, APM 데이터, 배포 diff

Stage 5: 진단
  → 가설 A 확인: 새 API 엔드포인트에서 N+1 쿼리 발견

Stage 6: 해결 설계
  옵션 1: eager loading 추가 (빠른 수정)
  옵션 2: 데이터 로더 패턴 도입 (근본 해결)
  추천: 옵션 1 즉시 + 옵션 2 후속 작업

Stage 7: 구현
  → eager loading 추가 + 테스트

Stage 8: 회고
  → PR 리뷰에 쿼리 카운트 체크 추가 제안
```

### 시나리오 2: 프로덕션 메모리 누수 디버깅

1. 메모리 사용량 그래프 분석, 증가 패턴 식별
2. 가설 수립 - 이벤트 리스너 미해제, 클로저 참조 누수, 캐시 무한 성장
3. 힙 스냅샷 비교로 누수 객체 유형 식별
4. WeakRef 미사용 캐시에서 참조 누수 확인
5. WeakMap 기반 캐시로 교체, TTL 정책 추가
6. 24시간 부하 테스트로 메모리 안정성 확인

## 메타인지 프롬프트

각 분석 단계에서 다음 자기 점검을 수행한다:

| 단계 | 자기 점검 질문 |
|------|---------------|
| 탐색 | 충분한 정보를 수집했는가? 빠뜨린 파일이 있는가? |
| 분석 | 내 가정은 검증되었는가? 반대 가설을 고려했는가? |
| 설계 | 더 단순한 해결책이 있는가? 과도 엔지니어링은 아닌가? |
| 구현 | 기존 코드 스타일과 일관적인가? 부작용은 없는가? |
| 검증 | 경계값 테스트를 했는가? 실패 모드를 확인했는가? |
