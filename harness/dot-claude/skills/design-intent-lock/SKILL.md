# Design Intent Lock — 의도 정합 앵커

> 디자인 작업의 첫 단추. "원하는 결과물"을 말로 끝내지 않고 구조로 고정해, 작업 내내 그 의도에서 벗어나지 않게 한다. 레퍼런스가 있으면 reference-match를 함께 쓴다(이 스킬=의도, reference-match=참조 재현).

## 언제 쓰나

화면·컴포넌트·랜딩·앱 UI 제작에 착수할 때. 이미 `design-brief.md`가 있으면 STEP 1을 건너뛰고 STEP 3(대조)부터.

## STEP 1 — 의도 수집 (AskUserQuestion 구조화)

아래 8축 중 **사용자가 이미 답을 준 축은 건너뛰고, 비어있는 축만** 묻는다. 추측으로 채우지 않는다.

| # | 축 | 확정할 것 |
|:-:|---|---|
| 1 | 목적 | 이 화면이 달성할 단 하나의 목표 (전환/정보전달/브랜딩/도구) |
| 2 | 타깃 | 사용자·맥락 (모바일 우선? 전문가? 신규 방문?) |
| 3 | 톤 & 무드 | 형용사 3개 (예: 미니멀·신뢰·차분 / 대담·역동·고급) |
| 4 | 레이아웃 의도 | 구조 (히어로+카드그리드 / 대시보드 / 폼 중심 / 피드) |
| 5 | 필수 요소 | 반드시 포함 (CTA·로고·특정 섹션·데이터·법적고지) |
| 6 | 제약 | 금지·고정 (브랜드 컬러 고정, 다크모드 필수, 스택 고정) |
| 7 | 레퍼런스 | 참조 자료 유무 — 있으면 reference-match로 라우팅 |
| 8 | 성공 기준 | "이렇게 되면 만족"의 검증 가능한 문장 |

톤·무드처럼 추상적인 축은 선택지(2~4개 무드 후보)를 제시해 고르게 한다.

## STEP 2 — design-brief.md 산출

프로젝트 루트 또는 `.thoughts/`에 작성한다:

```markdown
# Design Brief — <화면명> (YYYY-MM-DD)
- 목적: ...
- 타깃: ...
- 톤 & 무드: A / B / C
- 레이아웃 의도: ...
- 필수 요소: [ ] ...  [ ] ...
- 제약: ...
- 레퍼런스: <경로/URL 또는 없음>
- 성공 기준: ...

## 의도 변경 로그
- (brief를 바꿀 때마다 사유와 함께 append)
```

## STEP 3 — 작업 중 drift 대조

매 산출(컴포넌트·섹션 완성) 후 brief의 목적·톤·필수요소·제약과 대조한다.

- 이탈 발견 시: **"brief는 X인데 현재 Y"**를 명시하고 사용자에게 수정 또는 brief 갱신을 확인한다.
- 임의로 brief를 벗어나지 않는다. 더 나은 방향이 보이면 제안하되, **brief 갱신 승인 후** 진행하고 변경 로그에 남긴다.

## STEP 4 — 완료 검증

성공 기준 + 필수 요소 체크박스를 전수 대조한다. 미충족은 결함으로 보고하고, 충족 시 brief 대비 달성표를 출력한다.

## 출력 계약

- `design-brief.md` (필수 산출 — 세션 연속성 자산)
- 완료 시 brief 대비 충족표 (목적/톤/필수요소/성공기준 × 달성 여부)

## 연계

- 레퍼런스가 있으면 → **reference-match** (해부·대조)
- 코드 작업 중 → `rules/ui-ux-craft.md`가 상태·토큰·접근성을 상시 강제
- 라이브 검증 필요 → playwright-uiux-audit / playwright-design-audit
