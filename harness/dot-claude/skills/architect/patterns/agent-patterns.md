# Anthropic 에이전트 패턴 참조

> 출처: Anthropic, "Building Effective Agents" (2024)

---

## 6가지 기본 패턴

### 1. Prompt Chaining (프롬프트 체이닝)

**정의**: 작업을 순차적 단계로 분해하여, 각 단계의 출력이 다음 단계의 입력이 되는 구조.

**구조**:
```
[Step 1] ──출력──▶ [Step 2] ──출력──▶ [Step 3] ──▶ 최종 결과
```

**적용 시점**:
- 데이터 변환 파이프라인 (입력 → 정제 → 분석 → 출력)
- 단계별 검증이 필요한 워크플로
- 각 단계가 독립적 관심사를 가질 때

**Claude Code 매핑**: 파이프라인 레이어 간 데이터 흐름
- L0→L1→L2→L3 순차 처리
- 각 레이어가 명확한 입출력 타입 계약 보유

**주의점**:
- 단계가 많을수록 전체 지연시간 증가
- 중간 단계 실패 시 전체 파이프라인 중단 가능
- Gate/검증 단계를 중간에 삽입하여 조기 실패 처리

---

### 2. Routing (라우팅)

**정의**: 입력을 분류하여 적합한 처리 경로로 분기하는 구조.

**구조**:
```
           ┌──▶ [경로 A] ──▶ 결과 A
[입력] ──▶ [분류기] ──▶ [경로 B] ──▶ 결과 B
           └──▶ [경로 C] ──▶ 결과 C
```

**적용 시점**:
- 입력 유형에 따라 다른 처리 로직이 필요할 때
- 복잡도에 따라 경량/고급 처리를 구분할 때
- 도메인별 전문화된 처리가 필요할 때

**Claude Code 매핑**: 발화 유형별 분석 경로 분기
- 질문형 발화 → Q&A 분석기
- 결정형 발화 → 의사결정 추적기
- 일반 정보 → 요약기

**주의점**:
- 분류기의 정확도가 전체 시스템 품질 결정
- 분류 불가 입력에 대한 기본 경로(fallback) 필수
- 경로 수가 많을수록 유지보수 복잡도 증가

---

### 3. Parallelization (병렬화)

**정의**: 독립적인 작업을 동시에 실행하여 처리량을 높이는 구조.

**구조**:
```
         ┌──▶ [작업 A] ──┐
[입력] ──┼──▶ [작업 B] ──┼──▶ [결과 합성]
         └──▶ [작업 C] ──┘
```

**적용 시점**:
- 서로 독립적인 분석 작업 (예: 분류 + 요약 + 모순탐지)
- 여러 Worker가 동시에 다른 도메인 작업
- Evaluator들이 동시에 다른 관점에서 평가

**Claude Code 매핑**:
- Phase 1~4: Worker 4개 동시 작업 (각자 다른 도메인)
- Phase 5: Evaluator 3개 동시 평가 (QA + Security + Architect)
- 분석층: `Promise.all`로 분류/요약/모순탐지 병렬 실행

**주의점**:
- 작업 간 의존성이 있으면 병렬화 불가 (순차 처리로 전환)
- 결과 합성 로직이 복잡할 수 있음
- 자원 경합 주의 (같은 파일 동시 수정 금지)

---

### 4. Orchestrator-Workers (오케스트레이터-워커)

**정의**: 중앙 오케스트레이터가 작업을 분해하여 워커에게 위임하고 결과를 합성하는 구조.

**구조**:
```
                  ┌──▶ [Worker A] ──┐
[Orchestrator] ──┼──▶ [Worker B] ──┼──▶ [Orchestrator: 합성]
                  └──▶ [Worker C] ──┘
```

**적용 시점**:
- 복잡한 작업을 세분화하여 전문가에게 위임해야 할 때
- 작업 결과를 통합하여 일관된 산출물을 만들어야 할 때
- 작업 진행 상황을 중앙에서 모니터링해야 할 때

**Claude Code 매핑**: Lead → Worker 관계의 핵심 패턴
- Lead: Phase 관리, 작업 분배, 결과 통합, 품질 게이트
- Worker: 도메인별 자율 실행, 결과 보고

**주의점**:
- Orchestrator 병목 주의 (모든 것이 Lead를 통과)
- Worker 자율성과 Orchestrator 통제 사이 균형
- Worker 간 직접 통신 금지 (Orchestrator 경유)

---

### 5. Evaluator-Optimizer (평가자-최적화)

**정의**: 생성 결과를 평가하고, 기준 미달 시 재생성을 요청하는 피드백 루프.

**구조**:
```
[Generator] ──결과──▶ [Evaluator] ──판정──▶ 합격 → 출력
                         │                    불합격
                         └──피드백──▶ [Generator: 재생성]
```

**적용 시점**:
- 코드 품질 기준이 명확할 때
- 자동 테스트/검증이 가능할 때
- 반복적 개선으로 품질을 높일 수 있을 때

**Claude Code 매핑**: Worker-Evaluator 관계
- Worker가 코드 생성 → Evaluator가 검토
- 불합격 시 구체적 피드백과 함께 재작업 요청
- 최대 N회 재시도 (기본 2회) + 에스컬레이션

**주의점**:
- 평가 기준이 모호하면 무한 루프 위험
- 재작업 횟수 제한 필수 (기본 2회 권장)
- 에스컬레이션 정책: N회 초과 시 Lead → 사용자 판단

---

### 6. Context Augmentation (컨텍스트 보강)

**정의**: 외부 데이터를 동적으로 주입하여 처리 품질을 높이는 구조.

**구조**:
```
[입력] ──▶ [검색/수집] ──▶ [컨텍스트 결합] ──▶ [처리] ──▶ 결과
                ↑
           [외부 소스: DB, API, 문서]
```

**적용 시점**:
- 최신 정보가 필요한 분석 작업
- 외부 자료로 결과를 보강해야 할 때
- RAG(Retrieval-Augmented Generation) 패턴

**Claude Code 매핑**: 보강층 (Enrich Layer)
- 키워드 기반 외부 자료 검색
- 분석 결과에 참고 자료 첨부
- 리서치 모드 활성화 시 동작

**주의점**:
- 외부 소스의 신뢰성 검증 필요
- 토큰 한도 내에서 컨텍스트 양 조절
- 불필요한 보강은 노이즈 증가 → 선택적 활성화

## 하이브리드 조합 규칙

실제 프로젝트에서는 단일 패턴이 아닌 복수 패턴을 조합한다.

### 일반적 조합

| 조합 | 설명 | 적용 예 |
|------|------|---------|
| Orchestrator + Parallelization | 중앙 지시 + 병렬 실행 | Lead → Worker 4개 동시 |
| Orchestrator + Evaluator | 생성-평가 루프 | Worker 생성 → Evaluator 검토 |
| Chaining + Parallelization | 순차 파이프라인 내 병렬 단계 | L2→(L3 병렬 분석)→L4 |
| Routing + Parallelization | 분류 후 병렬 처리 | 발화 유형별 분기 + 동시 분석 |
| 전체 하이브리드 | 위 4가지 모두 조합 | voiceNote 아키텍처 |

### 복잡도별 권장 조합

| 복잡도 | 권장 패턴 | 에이전트 수 |
|:---:|------|:---:|
| 낮음 | Prompt Chaining 단독 | 1~2 |
| 중간 | Orchestrator + Parallelization | 3~5 |
| 높음 | Orchestrator + Parallelization + Evaluator | 5~10 |
| 매우 높음 | 전체 하이브리드 + Routing | 10+ |

### 패턴 선택 의사결정 트리

```
작업이 순차적인가?
├── Yes → Prompt Chaining
└── No → 독립적 병렬 작업이 가능한가?
    ├── Yes → Parallelization
    └── No → 중앙 조율이 필요한가?
        ├── Yes → Orchestrator-Workers
        └── No → 입력 유형에 따라 분기가 필요한가?
            ├── Yes → Routing
            └── No → 결과 품질 검증이 필요한가?
                ├── Yes → Evaluator-Optimizer
                └── No → 외부 정보가 필요한가?
                    ├── Yes → Context Augmentation
                    └── No → 단순 실행
```
