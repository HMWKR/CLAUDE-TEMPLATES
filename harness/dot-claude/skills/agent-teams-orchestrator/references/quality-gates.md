# 오케스트레이터 품질 게이트

> 이 파일은 agent-teams-orchestrator 스킬의 3단계 품질 게이트 체크리스트를 포함합니다.
> Lead가 각 게이트에서 이 파일을 참조하여 팀원 산출물을 검증합니다.

---

## 게이트 1: 팀원 작업 완료

팀원이 작업 완료를 보고하면 Lead가 검증:

### 체크리스트
- [ ] 요구사항 충족 여부
- [ ] 테스트 통과 여부
- [ ] 코드 품질 기준 충족
- [ ] 환각 방지 프로토콜 준수
  - 모든 코드/파일 참조에 [검증됨] 마커 있음
  - [추정] 항목에 대한 추후 검증 계획 명시
  - L1/L2 심각도 환각 없음 (→ exit code 2 재작업)
- [ ] 다른 팀원 영역 침범 없음

### 결과
- 통과 → 게이트 2로
- 미통과 → message로 수정 지시 (재작업)

Hook: TaskCompleted (exit code 2 = 재작업)

---

## 게이트 2: 통합 검증

모든 팀원 작업 완료 후 Lead가 통합 검증:

### 체크리스트
- [ ] 파일 충돌 없음
- [ ] 전체 테스트 스위트 통과
- [ ] 빌드 성공
- [ ] 팀원 간 인터페이스 정합성

### 결과
- 통과 → 게이트 3으로
- 미통과 → 관련 팀원에게 수정 지시

---

## 게이트 3: 최종 검수

domain-expert-analysis 검증 마커 적용:

### 환각 최종 검증 체크리스트
- [ ] 모든 팀원 산출물의 [검증됨] 마커 확인
- [ ] [문서확인] 항목의 실제 문서 소스 확인
- [ ] [추정] 항목 목록화 및 리스크 평가
- [ ] [미확인] 항목 해결 여부 확인
- [ ] 교차 팀원 검증 완료
- [ ] L1-L4 환각 심각도 검사 통과

### 검증 감사 로그 (Verification Audit Log)

```
[팀원명] [시각] [도구]: [대상] → [결과] [마커]
예: [Architect] Glob: utils/*.ts → 12 files [검증됨]
예: [Security] Read: engine.ts:245 → 확인 [검증됨]
예: [QA] [추정] 성능 영향: 테스트 필요
```

### 5단계 메타인지 자가 검증
1. **가정 검토**: 당연하게 여기는 전제는?
2. **반대 관점**: 이 결론에 반대하는 논거는?
3. **실패 모드**: 이 해결책이 실패할 시나리오는?
4. **숨겨진 변수**: 놓치고 있는 요소는?
5. **확신도**: 결론에 대한 확신도? (1-10)

### 결과
- 통과 → 사용자에게 최종 보고
- 미통과 → 해당 영역 재검수
