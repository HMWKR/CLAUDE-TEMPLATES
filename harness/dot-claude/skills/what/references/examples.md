# What 프레임워크 — 도메인별 예시

## 1. 코딩/개발

### 요청: "what 로그인 기능 추가해줘"

**STEP 1 — Why**:
| 프레임 | 내용 |
|--------|------|
| **Why** | 무단 접근을 방지하고 사용자별 개인화된 경험을 제공하기 위함 |

**STEP 2 — What**:
| 프레임 | 내용 |
|--------|------|
| **Why** | 무단 접근 방지 + 사용자별 개인화 |
| **What** | 이메일/비밀번호 기반 인증 시스템 구축 (회원가입, 로그인, 세션 관리) |

**STEP 3 — How**:
| 프레임 | 내용 |
|--------|------|
| **Why** | 무단 접근 방지 + 사용자별 개인화 |
| **What** | 이메일/비밀번호 기반 인증 시스템 |
| **How** | 1) DB 스키마 설계 → 2) API 엔드포인트 구현 → 3) JWT 토큰 관리 → 4) 프론트엔드 폼 → 5) 테스트 |

**STEP 4 — So What**:
| 프레임 | 내용 |
|--------|------|
| **Why** | 무단 접근 방지 + 사용자별 개인화 |
| **What** | 이메일/비밀번호 기반 인증 시스템 |
| **How** | DB → API → JWT → 프론트엔드 → 테스트 |
| **So What** | 사용자 데이터가 보호되고, 개인화된 기능(즐겨찾기, 히스토리)을 제공할 수 있게 됨. 성공 기준: 로그인/로그아웃 정상 동작 + 비인증 접근 차단 |

## 2. 기획/전략

### 요청: "what 경쟁사 분석 보고서 만들어줘"

**STEP 1 — Why**:
| 프레임 | 내용 |
|--------|------|
| **Why** | 경쟁사 분석 자체가 목적이 아니라, 자사 제품의 포지셔닝 전략을 수립하기 위한 근거가 필요 |

**STEP 2 — What**:
| 프레임 | 내용 |
|--------|------|
| **Why** | 자사 포지셔닝 전략 수립 근거 확보 |
| **What** | 상위 3개 경쟁사의 가격/기능/타겟 비교표 + 자사 차별화 포인트 도출 |

**STEP 3 — How**:
| 프레임 | 내용 |
|--------|------|
| **Why** | 자사 포지셔닝 전략 수립 근거 확보 |
| **What** | 경쟁사 비교표 + 차별화 포인트 |
| **How** | 1) 경쟁사 웹사이트/리뷰 분석 → 2) 가격/기능 매트릭스 작성 → 3) SWOT 분석 → 4) 차별화 전략 제안 |

**STEP 4 — So What**:
| 프레임 | 내용 |
|--------|------|
| **Why** | 자사 포지셔닝 전략 수립 근거 확보 |
| **What** | 경쟁사 비교표 + 차별화 포인트 |
| **How** | 웹 분석 → 매트릭스 → SWOT → 전략 제안 |
| **So What** | 다음 분기 제품 로드맵에 반영 가능한 구체적 차별화 방향이 확보됨. 성공 기준: 의사결정자가 "이걸로 방향 잡자"라고 말할 수 있는 수준 |

## 3. 분석/리서치

### 요청: "what 우리 서비스 이탈률 분석해줘"

**STEP 1 — Why**:
| 프레임 | 내용 |
|--------|------|
| **Why** | 이탈률 숫자 자체보다, 이탈의 근본 원인을 파악하여 리텐션을 개선하기 위함 |

**STEP 2 — What**:
| 프레임 | 내용 |
|--------|------|
| **Why** | 이탈 근본 원인 파악 → 리텐션 개선 |
| **What** | 이탈 시점별 원인 분류 + 상위 3개 원인에 대한 개선 방안 도출 |

**STEP 3 — How**:
| 프레임 | 내용 |
|--------|------|
| **Why** | 이탈 근본 원인 파악 → 리텐션 개선 |
| **What** | 이탈 원인 분류 + 개선 방안 |
| **How** | 1) 퍼널 데이터 수집 → 2) 이탈 시점 클러스터링 → 3) 사용자 세그먼트별 분석 → 4) 원인-개선안 매핑 |

**STEP 4 — So What**:
| 프레임 | 내용 |
|--------|------|
| **Why** | 이탈 근본 원인 파악 → 리텐션 개선 |
| **What** | 이탈 원인 분류 + 개선 방안 |
| **How** | 퍼널 → 클러스터링 → 세그먼트 → 매핑 |
| **So What** | 즉시 실행 가능한 리텐션 개선 액션 3개가 도출됨. 성공 기준: 각 액션에 담당자/일정/KPI가 부여될 수 있는 구체성 |

## 4. 문서/온보딩

### 요청: "what 우리 팀 온보딩 문서 만들어줘"

**STEP 1 — Why**:
| 프레임 | 내용 |
|--------|------|
| **Why** | 신규 입사자의 적응 기간을 단축하고, 팀 내 반복되는 온보딩 질문을 줄이기 위함 |

**STEP 2 — What**:
| 프레임 | 내용 |
|--------|------|
| **Why** | 적응 기간 단축 + 반복 질문 경감 |
| **What** | 신규 입사자가 첫 주 안에 독립적으로 개발 환경 세팅 + 첫 PR 제출까지 할 수 있는 가이드 |

**STEP 3 — How**:
| 프레임 | 내용 |
|--------|------|
| **Why** | 적응 기간 단축 + 반복 질문 경감 |
| **What** | 첫 주 독립 세팅 + 첫 PR 가이드 |
| **How** | 1) 환경 세팅 체크리스트 → 2) 코드베이스 구조 안내 → 3) 브랜치/PR 컨벤션 → 4) FAQ 정리 → 5) 멘토 연결 프로세스 |

**STEP 4 — So What**:
| 프레임 | 내용 |
|--------|------|
| **Why** | 적응 기간 단축 + 반복 질문 경감 |
| **What** | 첫 주 독립 세팅 + 첫 PR 가이드 |
| **How** | 체크리스트 → 구조 안내 → 컨벤션 → FAQ → 멘토 |
| **So What** | 온보딩 기간이 2주→1주로 단축되고, 시니어의 반복 설명 시간이 절감됨. 성공 기준: 신규 입사자가 문서만 보고 첫 PR을 제출할 수 있음 |

## 5. Backtrack 예시

### How 설계 중 What 수정이 필요한 경우

**상황**: STEP 3(How)에서 실행 단계를 분해하다가, STEP 2(What)에서 정의한 목표가 현실적으로 달성 불가능함을 발견

**대응**:
> "STEP 2(What)에서 '전체 DB 마이그레이션'을 목표로 정했지만, How를 설계하면서 현재 리소스로는 3개월 이상 소요됨을 확인했습니다. What의 범위를 조정해야 합니다."

→ AskUserQuestion:
- "이전 단계(What)로 돌아가서 범위 조정"
- "현재대로 진행 (리소스 추가 투입 전제)"

→ "이전 단계로" 선택 시: STEP 2 Confirmation Loop 재진입, STEP 3 내용 리셋

## 6. 실행 방식 선택 예시

4단계 모두 확정 후:

> **최종 요약**
> | 프레임 | 내용 |
> |--------|------|
> | Why | 적응 기간 단축 + 반복 질문 경감 |
> | What | 첫 주 독립 세팅 + 첫 PR 가이드 |
> | How | 체크리스트 → 구조 안내 → 컨벤션 → FAQ → 멘토 |
> | So What | 온보딩 2주→1주, 시니어 시간 절감 |
>
> "이 방향으로 작업을 시작하겠습니다."

→ AskUserQuestion: 실행 방식
- "TodoWrite로 단계별 추적하며 실행" → How의 5단계가 TodoWrite 항목으로 변환
- "바로 실행" → 즉시 작업 수행
