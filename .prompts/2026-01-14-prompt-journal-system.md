# 프롬프트 저널 시스템 설계

## 원본 프롬프트

> 16섹션 커밋 방식에 대해 의구심이 드는데, 더 나은 대안이 있을까? A(연구 데이터 수집), B(개인 학습/회고), C(딥러닝 모델 학습)를 모두 달성하고 싶어.

## 사고 여정

### 처음 생각
처음에는 16섹션 커밋을 완전히 대체하는 방안을 생각했다.
"커밋 메시지가 너무 길다" → "간결하게 만들자" → "16섹션 강제 해제"

이건 기존 시스템을 버리는 급진적인 변화였다.

### 전환점 1: 사용자의 피드백

사용자가 중요한 피드백을 주었다:
> "텍스트로 하는 작업은 필요해보여. 내가 그걸 토대로 더 나은 프롬프트를 작성할 수 있으니."

이 순간 깨달았다:
- JSON만으로는 부족하다
- 사람이 **읽고 배울 수 있어야** 한다
- 커밋 메시지가 아닌 **별도의 문서**가 필요하다

### 전환점 2: 병행 운영 선택

"16섹션 강제 해제"를 계획에 넣었는데, 사용자가 질문했다:
> "16섹션 강제 해제는 왜 넣은지?"

이 질문을 받고 깨달았다:
- 내가 암묵적으로 "대체"를 가정했다
- 하지만 사용자는 "병행"을 원할 수도 있다
- 가정하지 말고 물어봐야 한다

결과: 사용자가 "병행 운영"을 선택
- 16섹션 커밋 유지
- 프롬프트 저널 추가

### 최종 통찰

**듀얼 포맷 시스템**이 답이다:
1. 16섹션 커밋: Git 히스토리에 상세 정보 (기존 유지)
2. 프롬프트 저널: 읽고 배우기 위한 마크다운 문서 (신규 추가)
3. 두 소스에서 prompts.json 생성 → 더 풍부한 데이터

핵심 원칙:
- 사람이 읽을 수 있어야 한다 (마크다운)
- 기계가 파싱할 수 있어야 한다 (YAML frontmatter)
- 기존 시스템을 버리지 않는다 (병행)

## 핵심 학습

1. **가정하지 말고 물어보라**
   - "16섹션 해제"를 당연하게 넣었다가 질문 받음
   - 사용자의 의도는 내 가정과 다를 수 있다
   - 중요한 결정은 확인하자

2. **텍스트의 가치**
   - JSON은 기계가 읽기 좋다
   - 하지만 사람이 읽고 배우려면 텍스트/마크다운이 필요
   - 둘 다 제공하는 것이 최선 (듀얼 포맷)

3. **기존 시스템의 가치**
   - 새로운 것을 추가한다고 기존 것을 버릴 필요 없다
   - 병행 운영이 더 풍부한 데이터를 만든다
   - 진화, not 혁명

4. **프롬프트 저널의 본질**
   - 개발자의 일기처럼
   - "오늘 어떤 프롬프트를 작성했는가?"
   - "어떤 사고 과정을 거쳤는가?"
   - "무엇을 배웠는가?"
